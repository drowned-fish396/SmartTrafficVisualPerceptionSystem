# AGENTS.md

## 项目定位

本项目是软件工程课程项目，主题为“云边端协同的智慧交通视觉感知系统”。系统需要模拟一个由手机、云服务器、前端展示端共同组成的智慧交通视觉感知平台，用于完成交通场景视频采集、传输、实时视频流AI 分析、可视化展示、系统管理、设备管理、模型管理、监控展示和历史查询。

项目应优先面向学院 704 智慧交通沙盘场景，也可以兼容校内外道路场景。实现时要突出“云边端协同”这一核心特征，避免把项目做成单机脚本或纯前端展示页面。

## 核心场景

- 边端采集设备：使用手机采集交通视频，负责推流、上传视频，必要时也可承担轻量分析任务。
- 云端分析服务：使用电脑作为云服务器，负责拉流或接收视频，并运行 AI 检测、识别、跟踪、告警和统计逻辑。
- 前端展示端：使用另一台电脑或浏览器前端展示实时分析结果、系统状态、设备状态、模型配置和历史数据。
- 交通场景：首选 704 智慧交通沙盘，也可扩展到校园道路、校外道路等真实交通画面。

## 必做功能方向

实现功能时优先围绕以下模块拆分需求、接口和页面。

### 视觉感知算法(关键)

- 车牌识别：对视频流中的车辆进行实时车牌检测与识别，将识别结果与本地白名单车辆库比对，输出是否允许通行的决策结果。课程不要求真实控制闸机。
- 车辆识别与拥堵热力图：检测画面中的车辆目标，按区域统计车辆数量和密度，动态生成拥堵热力图，展示交通拥堵态势。
- 车辆跟踪：对进入禁停区域的车辆持续跟踪，计算区域内停留时长，超过预设阈值后触发长时间停留告警。
- 道路异常检测：检测路面中不应存在的异常物体，发现后即时告警，并标注异常位置和受影响车道。

### 平台管理功能

- 系统管理：监控 CPU、GPU、内存使用率，以及拉流、推流、视频传输和分析任务状态。
- 设备管理：支持边缘设备注册、设备列表展示、在线状态监控、最近心跳时间和视频源状态展示。
- 模型管理：支持模型选择、参数配置、阈值配置、禁停区域配置、检测区域配置和模型运行状态展示。
- 监控展示：提供闸机监控、拥堵热力图、禁停区域监控、道路异常监控等展示视图，支持通过移动手机位置切换监控画面、模型和展示方式。
- 数据统计和历史查询：展示检测车牌数量、拥堵度变化趋势、违规停车数量、道路异常数量和异常位置分布，支持按时间范围查询。

## 建议系统结构

后续实现时优先采用清晰的三端结构，接口边界要明确。

- edge：手机或边缘采集端相关代码，包括视频采集、推流、上传、设备心跳、设备注册等。
- backend：云端服务相关代码，包括视频接入、AI 推理、事件生成、设备管理、模型管理、系统监控、历史数据存储和 API。
- frontend：前端展示端相关代码，包括实时监控大屏、管理页面、模型配置页面、统计查询页面和告警展示。
- docs：课程成果物文档，包括需求分析报告、系统设计方案、软件说明书、系统测试报告、工作日报和总结报告。
- data：示例视频、白名单车辆库、测试数据、历史统计样例等。不要提交过大的原始视频文件，必要时使用说明文档标注获取方式。
- models：模型文件、模型配置或模型下载说明。大模型文件不要直接提交到仓库，使用下载脚本或 README 说明。
- tests：后端接口测试、算法模块测试、前端关键流程测试和系统联调测试。

## 工程要求

- 任何实现都应围绕课程评分标准服务：需求分析准确性、系统方案合理性、系统实现完备性、系统实现效果、系统创新性、团队合作效率和成果物可读性。
- 优先保证可演示闭环，再扩展高级功能。不要只写离线算法脚本而缺少前端展示和系统管理能力。
- 云端、边端、前端之间的数据协议要明确，建议使用 REST API、WebSocket、RTSP/RTMP/HTTP 视频流或文件上传中的一种或多种组合。
- 检测、识别、跟踪和告警结果应使用结构化数据表达，至少包含时间、来源设备、任务类型、目标位置、置信度、事件类型和可视化所需字段。
- 模型推理代码应允许替换模型或模拟模型输出，便于在没有 GPU 或模型文件不完整时仍能完成系统演示。
- 前端页面应服务于监控和管理场景，避免只做营销页。信息展示要清晰、实时、可解释。
- 历史数据、白名单车辆库、设备列表和模型配置应尽量持久化，便于演示查询和复现。
- 对外部依赖、模型文件、启动命令、端口、测试账号和演示步骤要写入文档。

## 开发协作规则

- 决策时优先分析上下文，测试和配置
- 开始写代码前，先确认当前目标属于需求文档、系统设计、后端、前端、算法、测试还是部署说明。
- 不要猜测我的意图,不要假设我知道自己想要什么，任何不明确的需求、改动或目标都一定要停下，向我确认和讨论
- 改动前先查看现有目录和 README，不要假设项目结构已经存在。
- 优先做小而完整的改动，每次改动都要能解释它服务于哪一项课程要求。
- 任何改动都要从根本上解决，不能只改动表面
- 改动某一个模块的bug时，先复述当前bug，之后告诉我会影响那些地方，确定bug出现的可能原因，修改时要说明后续如何验证
- 如果需要使用真实视频或模型权重，优先提供下载说明、占位文件或 `.gitignore` 规则。
- 新增接口时同步更新接口文档或软件说明书。
- 新增页面时同步说明其演示用途和对应课程功能点。
- 新增算法模块时同步提供输入、输出、阈值、性能假设和无模型环境下的降级方案。

## 推荐演示主线

验收演示建议按以下顺序组织：

1. 手机作为边端设备采集或上传交通视频。
2. 云端服务接收视频并显示设备在线、视频传输状态和系统资源状态。
3. 选择或切换 AI 模型，展示车辆检测、车牌识别、拥堵热力图、禁停跟踪或道路异常检测。
4. 前端实时展示检测框、车牌结果、热力图、告警信息和统计指标。
5. 查询历史数据，展示车牌数量、拥堵趋势、违规停车数量、异常数量和位置分布。
6. 说明系统架构、模块分工、测试结果和可扩展方向。
