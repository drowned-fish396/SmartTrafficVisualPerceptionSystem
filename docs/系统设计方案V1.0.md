# 云边端协同的智慧交通视觉感知系统系统设计方案 V1.0

## 1. 引言

### 1.1 编写目的

本文档用于在《云边端协同的智慧交通视觉感知系统需求分析报告 V1.0》的基础上，进一步明确系统的总体架构、功能架构、技术架构、模块划分、数据结构、接口通信、部署运行和测试验收设计。

本文档面向课程评审和项目开发两类读者。一方面，文档需要说明系统设计是否能够覆盖课程项目要求，是否体现“云边端协同”的核心特征，是否具备可演示和可扩展的工程方案；另一方面，文档也需要为后续代码实现提供模块边界、接口边界、数据边界和部署边界。

系统设计方案 V1.0 阶段不追求生产级完整实现，而是优先保证系统能够形成“边端采集、云端分析、前端展示、历史查询”的演示闭环，并为后续 V2.0 设计、系统实现、系统测试和课程验收提供依据。

### 1.2 项目背景

本项目是软件工程课程项目，主题为“云边端协同的智慧交通视觉感知系统”。系统面向学院 704 智慧交通沙盘场景，也可兼容校园道路、校外道路等真实交通画面，用于模拟智慧交通平台中的视频采集、视频传输、实时 AI 分析、可视化展示、系统管理、设备管理、模型管理和历史查询流程。

智慧交通视觉感知场景中，车牌识别、车辆检测、拥堵分析、禁停区域跟踪和道路异常检测是典型应用任务。传统离线视频分析方式难以体现真实系统中的设备接入、状态监控、视频传输、云端计算和前端展示链路，因此本项目采用云边端协同思路，将手机作为边端采集设备，将电脑作为云端分析服务器，将浏览器或另一台电脑作为前端展示端，形成三端协作的系统结构。

系统设计 V1.0 的核心目标是构建一个可落地、可演示、可扩展的云边端协同视觉感知平台，使系统能够完成从手机视频采集到云端 AI 分析，再到前端实时展示和历史查询的完整链路。

系统设计目标包括以下四类：

| 约束类型 | 具体约束 | 设计影响 |

## 3. 架构风格与设计原则

### 3.1 架构风格选择

本系统 V1.0 采用“云边端协同的分层模块化架构”。

从系统整体看，边端手机负责交通场景视频采集、实时推流或视频上传、设备注册和设备心跳上报；云端服务负责视频接入、AI 推理、业务规则处理、数据存储、状态监控和实时消息推送；前端展示端负责实时监控、设备管理、模型管理、系统状态展示和历史查询。

从云端内部看，后端服务采用模块化单体设计，将接口层、业务服务层、算法推理层、数据访问层和基础设施层进行逻辑拆分。V1.0 阶段各模块可以部署在同一后端服务中，降低开发和部署复杂度；后续如果系统规模扩大，可以将视频处理、AI 推理、实时推送等模块逐步拆分为独立服务。

### 3.2 架构分层设计

系统按照职责划分为以下层次：

| 层次 | 主要内容 | 作用 |
|---|---|---|
| 边端采集层 | 手机摄像头、推流工具、上传页面、设备心跳 | 负责现场视频输入和设备状态上报 |
| 接入通信层 | REST API、WebSocket、RTSP/RTMP、文件上传 | 负责三端之间的数据交换 |
| 云端业务层 | 设备管理、视频源管理、任务管理、模型配置、历史查询 | 负责系统业务逻辑处理 |
| 算法分析层 | 车辆检测、车牌识别、拥堵分析、禁停跟踪、异常检测 | 负责从视频帧中生成结构化分析结果 |
| 数据支撑层 | SQLite/PostgreSQL、视频文件、模型配置、白名单数据 | 负责持久化数据和配置管理 |
| 前端展示层 | 监控大屏、设备管理、模型管理、系统状态、历史查询页面 | 负责用户交互和可视化展示 |

### 3.3 设计原则

系统设计遵循以下原则：

| 设计原则 | 说明 |
|---|---|
| 演示闭环优先 | 优先保证采集、传输、分析、展示和查询流程完整可演示 |
| 云边端职责清晰 | 边端、云端、前端分别承担明确职责，通过协议交互 |
| 模块化设计 | 各功能模块保持清晰边界，便于分工开发和后续维护 |
| 可降级运行 | 实时流不可用时支持视频上传，模型不可用时支持模拟结果 |
| 数据结构化 | 设备状态、检测结果、告警事件、统计数据和模型配置统一结构化表达 |
| 可扩展 | 预留多设备、多模型、多场景和数据库升级能力 |
| 可测试 | 每个模块具备明确输入、输出和可验证结果 |

### 3.4 V1.0 架构取舍

系统 V1.0 不采用完整微服务架构，而采用模块化单体后端方案。该方案能够在课程项目周期内降低部署和联调成本，同时保留模块拆分边界，便于后续演进。

系统 V1.0 不将 AI 推理逻辑写成完全独立的离线脚本，而是将算法推理作为云端业务流程的一部分，使分析结果能够被前端实时展示、被数据库持久化，并被历史查询模块复用。

系统 V1.0 优先使用真实模型完成车辆检测、车牌识别、拥堵分析、禁停跟踪和道路异常检测；当模型文件缺失、模型加载失败、识别效果不稳定或硬件性能不足时，算法模块应支持输出结构化模拟结果，作为演示兜底机制，保证云端分析、前端展示和历史记录流程完整可用。

## 4. 系统总体架构设计

### 4.1 系统总体组成

系统由边端采集端、云端分析服务、前端展示端、数据存储和模型资源组成。

| 组成部分 | 角色定位 | 主要职责 |
|---|---|---|
| 边端采集端 | 手机或移动采集设备 | 采集 704 智慧交通沙盘视频，进行实时推流或视频上传，上报设备状态 |
| 云端分析服务 | 后端服务和 AI 分析中心 | 接收视频输入，执行 AI 分析，生成检测结果、统计结果和告警事件 |
| 前端展示端 | 浏览器监控和管理界面 | 展示实时视频、分析结果、系统状态、设备状态、模型配置和历史数据 |
| 数据存储 | 数据库和文件存储 | 保存设备、任务、配置、检测结果、告警事件、统计数据和视频文件信息 |
| 模型资源 | 模型文件和推理配置 | 支撑车辆检测、车牌识别、拥堵分析、禁停跟踪和异常检测 |

### 4.2 系统总体架构说明

系统运行时，手机边端采集交通视频，并通过实时视频流或视频文件上传方式将视频输入传输至云端。云端服务接收视频源后创建分析任务，通过 OpenCV 或 FFmpeg 等工具进行视频读取和抽帧处理，再调用视觉感知算法模块生成车辆检测、车牌识别、拥堵状态、禁停告警和道路异常等结构化结果。

云端服务将实时分析结果通过 WebSocket 推送给前端展示端，前端在实时监控页面中展示视频画面、检测框、车牌识别结果、拥堵热力图和告警信息。同时，云端服务将关键检测结果、告警事件和统计数据写入数据库，供历史查询和统计分析页面使用。

前端展示端还可以通过 REST API 向云端提交设备管理、模型配置、阈值配置、区域配置和历史查询请求。云端服务根据前端配置更新分析任务参数，使系统具备基本管理能力。

### 4.4 系统核心链路

系统核心链路包括三条：

| 链路 | 方向 | 数据内容 | 通信方式 |
|---|---|---|---|
| 视频输入链路 | 边端采集端到云端分析服务 | 实时视频流或视频文件 | RTSP/RTMP 或 HTTP 文件上传 |
| 实时结果链路 | 云端分析服务到前端展示端 | 检测结果、车牌结果、热力图、告警事件、系统状态 | WebSocket |
| 管理查询链路 | 前端展示端到云端分析服务 | 设备查询、模型配置、区域配置、历史查询 | REST API |

云端到边端的采集参数下发或设备控制指令可作为 P2 扩展链路保留，V1.0 阶段不作为核心演示链路强制实现。

## 5. 功能架构设计

### 5.1 功能架构划分思路

系统功能架构按照用户可感知能力和业务职责进行划分，不直接等同于代码目录结构。功能架构需要体现系统从边端采集、云端分析到前端展示和历史查询的完整业务链路，同时覆盖设备管理、模型管理和系统管理等平台能力。

V1.0 阶段功能架构划分为五个层次：边端接入层、云端业务层、算法能力层、前端展示层和数据支撑层。边端接入层负责采集和上传视频，云端业务层负责视频接入、任务管理和平台业务处理，算法能力层负责视觉感知分析，前端展示层负责监控和管理页面，数据支撑层负责保存设备、配置、结果、告警和统计数据。

| 模型管理模块 | 管理模型选择、阈值配置、检测区域、禁停区域和运行状态 | 模型管理 |
| 系统管理模块 | 监控 CPU、GPU、内存、视频传输状态和分析任务状态 | 系统管理 |
| 实时监控模块 | 展示实时视频、检测框、车牌结果、拥堵热力图和告警信息 | 监控展示 |
| 历史查询模块 | 查询车牌记录、拥堵统计、禁停事件和道路异常事件 | 数据统计和历史查询 |
| 告警管理模块 | 生成、展示和查询禁停告警、道路异常告警等事件 | 告警展示和历史追溯 |
| 数据支撑模块 | 保存设备、视频源、任务、配置、检测结果、告警事件和统计数据 | 数据持久化 |

### 5.4 边端采集功能

边端采集功能面向手机或移动采集设备，负责为系统提供交通场景视频输入。V1.0 阶段边端采集功能应支持两种输入方式：实时视频流和短视频上传。实时视频流用于演示系统实时分析能力，短视频上传用于在网络或推流环境不稳定时保证系统演示可继续进行。

边端采集功能还需要支持设备注册和心跳上报。设备注册用于建立边端设备与云端系统之间的身份关系，心跳上报用于维护设备在线状态和最近活动时间。

### 5.5 云端业务功能

云端业务功能是系统的核心处理部分，负责连接边端输入、算法分析、数据存储和前端展示。云端业务功能包括视频接入、分析任务管理、设备管理、模型管理、系统监控、历史查询和实时推送。

视频接入功能负责将实时视频流或上传视频文件转换为可分析的视频源。分析任务功能负责对视频源创建分析任务，并协调抽帧、算法推理、结果汇总和状态更新。设备管理、模型管理和系统管理功能负责支撑平台管理能力，使系统不只具备算法演示能力，也具备基本运行管理能力。

### 5.6 算法分析功能

算法分析功能负责从视频帧中提取交通场景信息。V1.0 阶段算法功能包括车辆检测、车牌识别、白名单比对、拥堵热力图生成、禁停区域车辆跟踪和道路异常检测。

算法分析结果需要统一输出为结构化数据。结构化结果既用于前端实时可视化，也用于数据库持久化和历史查询。对于模型不可用或识别效果不稳定的情况，算法模块需要支持结构化模拟结果输出，保证系统链路完整。

### 5.7 前端展示功能

前端展示功能面向监控人员、平台管理员和模型配置人员。V1.0 阶段前端应包含实时监控大屏、设备管理页面、模型管理页面、系统状态页面和历史查询页面。

实时监控大屏用于展示视频画面、检测框、车牌识别结果、拥堵热力图和告警列表。设备管理页面用于展示边端设备在线状态、最近心跳和视频源状态。模型管理页面用于展示模型运行状态和配置参数。系统状态页面用于展示云端资源和任务状态。历史查询页面用于查询检测记录、告警事件和统计趋势。

### 5.8 数据支撑功能

数据支撑功能为系统运行和历史查询提供基础。V1.0 阶段需要保存设备信息、视频源信息、分析任务信息、模型配置信息、白名单车辆库、检测结果、车牌记录、拥堵统计、禁停事件、道路异常事件和系统状态数据。

数据支撑功能需要同时服务实时展示和历史查询。实时展示更关注最近一段时间的结果推送，历史查询更关注持久化记录、时间范围筛选和统计结果复现。

## 6. 技术架构设计

### 6.1 技术架构设计目标

技术架构设计需要支撑系统的三端协同、实时分析、可视化展示和历史查询能力。V1.0 阶段技术选型应优先满足可实现、可演示、易部署、易调试和可扩展要求，避免引入过重的基础设施。

系统技术架构围绕“手机边端 + Python 云端服务 + Web 前端 + 数据库 + 模型资源”构建。边端通过实时视频流或文件上传提供视频输入，云端通过 FastAPI 提供 REST API 和 WebSocket 通信能力，通过 OpenCV/FFmpeg 处理视频帧，通过视觉算法模块生成分析结果，通过数据库保存业务数据。前端通过 Vue 3 和 ECharts 实现实时监控、管理配置和统计展示。

后端采用模块化单体结构，内部按照设备管理、视频接入、分析任务、模型配置、系统监控、历史查询和实时推送进行模块划分。

### 6.5 视频接入技术

视频接入采用实时流和文件上传两种方式。

| 输入方式 | 说明 | V1.0 定位 |
|---|---|---|
| RTSP/RTMP 实时流 | 手机或推流工具将实时视频传输至云端，云端读取视频流并抽帧分析 | 主方案 |
| HTTP 视频上传 | 手机或前端上传短视频文件，云端保存后进入分析流程 | 兜底方案 |

实时流方案用于体现系统实时性，视频上传方案用于保证演示环境不稳定时仍可完成分析流程。云端视频处理模块应尽量复用同一套抽帧和分析逻辑，使实时流和上传视频能够进入统一分析任务流程。

### 6.6 AI 算法技术

视觉感知算法建议采用 YOLO 目标检测、PaddleOCR 字符识别、ByteTrack 目标跟踪、FastFlow 异常检测、区域规则和模拟结果兜底相结合的方式。

| 算法能力 | 技术路线 | 输出结果 |
|---|---|---|
| 车辆检测 | YOLO 系列轻量检测模型 | 车辆框、类别、置信度 |
| 车牌识别 | YOLO 车牌定位 + PaddleOCR 字符识别 + 车牌规则校验 | 车牌号、车牌框、识别置信度、白名单比对结果 |
| 拥堵分析 | YOLO 车辆检测 + 区域密度规则 + 时间窗口统计 | 区域车辆数量、车辆密度、拥堵等级、热力图数据 |
| 禁停跟踪 | YOLO 车辆检测 + ByteTrack 跟踪 + 禁停区域规则 | 目标 ID、停留时长、禁停告警 |
| 道路异常检测 | FastFlow 异常检测模型 + 规则检测兜底 | 异常区域、异常分数、影响车道、告警等级 |
| 模拟结果兜底 | 预设样例数据、固定轨迹、固定事件 | 结构化演示结果 |

V1.0 阶段算法重点是输出稳定、结构统一、可被前端展示和数据库保存的结果。车牌识别不引入其他 OCR 主方案，正式技术路线以 YOLO + PaddleOCR 为准；道路异常检测不将常规目标检测作为主方案，而以 FastFlow 异常检测模型识别异常区域，并通过规则检测保证演示可用性。算法准确率目标限定在课程演示场景，不承诺生产级复杂道路环境准确率。

### 6.7 数据存储技术

V1.0 阶段建议使用 SQLite 作为默认数据库，降低部署复杂度。后续如果系统扩展到多设备持续接入、多任务并行和大量历史数据查询，可以切换为 PostgreSQL。

数据库主要保存设备信息、视频源信息、分析任务、模型配置、检测结果、车牌记录、白名单车辆、拥堵统计、禁停事件、异常事件、系统状态和操作日志等数据。

视频文件、模型文件和示例数据不建议直接全部写入数据库。视频文件和模型文件可保存在文件系统中，数据库只保存文件路径、文件状态、关联任务和元数据。

### 6.8 通信协议技术

系统通信协议设计如下：

| 通信方向 | 数据内容 | 推荐方式 |
|---|---|---|
| 边端到云端 | 实时视频流 | RTSP/RTMP |
| 边端到云端 | 设备注册、心跳、视频上传 | REST API、HTTP 文件上传 |
| 云端到前端 | 检测结果、告警事件、任务状态、系统状态 | WebSocket |
| 前端到云端 | 设备查询、模型配置、区域配置、历史查询 | REST API |

REST API 用于请求响应型业务，WebSocket 用于实时推送型业务，视频流协议用于视频传输。三类通信方式分别服务不同数据特点，避免将所有数据都塞入同一种协议。

## 7. 云边端协同机制设计

### 7.1 云边端职责划分

系统采用明确的三端职责划分，使边端、云端和前端之间通过协议协作，而不是互相承担对方职责。

| 端 | 主要职责 | 不承担的职责 |
|---|---|---|
| 边端采集端 | 采集视频、实时推流、上传视频、注册设备、上报心跳 | 不承担重型 AI 推理和历史数据管理 |
| 云端分析服务 | 接收视频、创建任务、执行 AI 分析、保存数据、推送结果、管理配置 | 不直接承担复杂用户界面展示 |
| 前端展示端 | 实时监控、设备管理、模型配置、系统状态展示、历史查询 | 不直接执行视频推理和数据库写入 |

### 7.2 边端到云端协同

边端到云端协同包括视频输入协同和设备状态协同。

视频输入协同指边端通过实时视频流或视频上传方式向云端提供交通场景视频。云端需要为不同视频输入建立视频源记录，并将视频源关联到具体设备和分析任务。

设备状态协同指边端通过设备注册和心跳上报机制向云端报告自身状态。云端根据最近心跳时间判断设备是否在线，并在设备管理页面展示设备状态、视频源状态和当前任务状态。

### 7.3 云端内部协同

云端内部通过分析任务连接视频接入、算法推理、数据存储和实时推送。视频接入模块获取视频帧后，将帧数据或帧路径交给算法分析模块。算法分析模块输出结构化结果后，分析任务模块负责汇总结果、更新任务状态、触发告警逻辑，并将需要持久化的数据写入数据库。

实时结果由 WebSocket 推送模块发送至前端，历史结果由数据访问模块保存到数据库。模型配置、检测区域和禁停区域由模型管理模块维护，并在分析任务运行时提供给算法模块使用。

### 7.4 云端到前端协同

云端到前端协同主要通过 WebSocket 和 REST API 完成。

WebSocket 用于推送实时数据，包括车辆检测框、车牌识别结果、拥堵热力图数据、禁停告警、道路异常告警、分析任务状态和系统资源状态。前端收到消息后更新监控页面，实现近实时展示。

REST API 用于前端主动请求数据或提交配置，包括查询设备列表、查询历史记录、保存模型阈值、保存检测区域、保存禁停区域和查询系统状态等。

### 7.5 前端到云端协同

前端到云端协同主要体现为管理配置和历史查询。模型配置人员可以通过前端调整模型阈值、检测区域和禁停区域，云端保存配置后供分析任务使用。监控查看人员可以通过前端按时间范围、设备、事件类型查询历史数据，云端从数据库读取并返回结果。

前端提交的配置不直接操作算法内部实现，而是通过云端配置管理模块统一保存和下发给分析任务。这样可以保证配置来源可追踪，避免前端与算法模块强耦合。

### 7.6 协同数据类型

系统协同过程中涉及以下主要数据类型：

| 数据类型 | 产生位置 | 消费位置 | 说明 |
|---|---|---|---|
| 设备数据 | 边端采集端 | 云端、前端 | 包括设备编号、设备名称、在线状态和心跳时间 |
| 视频源数据 | 边端采集端、云端 | 云端、前端 | 包括实时流地址、上传视频路径、视频源状态 |
| 分析任务数据 | 云端 | 云端、前端 | 包括任务编号、任务状态、开始时间、结束时间 |
| 检测结果数据 | 算法模块 | 云端、前端、数据库 | 包括目标类别、位置框、置信度和时间戳 |
| 告警事件数据 | 云端业务模块 | 前端、数据库 | 包括告警类型、告警等级、位置和处理状态 |
| 配置数据 | 前端、云端 | 云端、算法模块 | 包括模型阈值、检测区域、禁停区域和白名单 |
| 系统状态数据 | 云端 | 前端、数据库 | 包括 CPU、GPU、内存、任务数量和视频传输状态 |

### 7.7 协同异常处理

云边端协同过程中需要考虑基础异常处理。

| 异常情况 | 处理方式 |
|---|---|
| 边端心跳超时 | 将设备状态标记为离线，并在设备管理页面提示 |
| 视频流中断 | 更新视频源状态，保留任务记录，提示用户切换视频上传方式 |
| WebSocket 断开 | 前端尝试重连，云端保留最近状态供 REST API 查询 |
| 模型加载失败 | 记录错误状态，切换到模拟推理或提示模型不可用 |
| 数据库写入失败 | 记录日志并返回错误状态，避免前端误判任务成功 |

V1.0 阶段异常处理重点是让系统状态可见、错误原因可解释，并保留演示兜底路径。

## 8. 后端服务模块设计

### 8.1 后端服务定位

后端服务是系统的云端核心，负责承接边端视频输入、组织 AI 分析流程、管理设备与模型配置、保存业务数据，并向前端提供 REST API 和 WebSocket 实时推送能力。

V1.0 阶段后端采用模块化单体结构。各业务模块部署在同一个 FastAPI 服务中，但在代码结构和职责上保持清晰边界，便于后续按需要拆分为独立服务。

| 数据访问模块 | 封装数据库读写操作 | 业务数据对象 | 数据库记录 |

### 8.4 设备管理服务设计

设备管理服务负责维护边端设备信息。边端设备首次接入时向云端提交设备名称、设备类型、视频源信息等注册数据，云端生成或绑定设备编号，并保存设备记录。

设备管理服务应支持心跳更新。边端定期上报心跳和视频源状态，云端根据最近心跳时间判断设备是否在线。当设备超过设定时间未上报心跳时，系统应将设备状态标记为离线，并在前端设备管理页面提示。

设备数据至少包含设备编号、设备名称、设备类型、注册时间、在线状态、最近心跳时间、视频源地址和当前任务状态。

### 8.5 视频接入服务设计

视频接入服务负责统一管理实时视频流和上传视频文件。对于实时流，云端保存视频流地址和连接状态，并通过 OpenCV 或 FFmpeg 拉取视频帧。对于上传视频，云端保存视频文件路径、上传时间和关联设备，并将其作为分析任务输入。

实时流和上传视频应尽量进入统一的分析任务流程，避免形成两套割裂的处理逻辑。视频接入服务需要维护视频源状态，包括连接中、分析中、中断、结束和异常等状态。

### 8.6 分析任务服务设计

分析任务服务负责将视频源、模型配置和算法模块组织成完整分析流程。任务创建时需要记录任务编号、设备编号、视频源编号、任务类型、启动时间、任务状态和配置参数。

分析任务运行过程中，服务按照配置频率抽取视频帧，并调用车辆检测、车牌识别、拥堵分析、禁停跟踪和道路异常检测模块。分析结果生成后，任务服务负责结果汇总、告警判断、数据落库和实时推送。

任务状态应至少包括待启动、运行中、已暂停、已结束和异常。任务异常时应保存错误原因，便于前端展示和后续排查。

### 8.7 模型配置服务设计

模型配置服务负责维护 AI 分析相关配置，使模型运行参数能够通过前端管理页面调整。V1.0 阶段不要求在线训练模型，但应支持真实模型模式和模拟结果兜底模式之间切换。

模型参数应包括模型名称、模型类型、模型文件路径、启用状态、运行状态、推理频率、置信度阈值、异常分数阈值、检测区域、禁停区域、拥堵阈值、停留时间阈值和是否启用模拟模式。

配置变更后，云端应保存配置记录，并在分析任务启动或刷新配置时读取最新参数。对于正在运行的任务，V1.0 可优先采用“保存配置后重新启动任务生效”的方式，降低实时热更新复杂度。

### 8.8 系统监控服务设计

系统监控服务负责采集云端服务运行状态，包括 CPU 使用率、内存使用率、GPU 使用率、视频传输状态、拉流状态、分析任务状态和 WebSocket 连接状态。

如果本地环境无法获取 GPU 信息，系统应显示“未检测到 GPU”或“GPU 监控不可用”，不影响视频接入、AI 分析和前端展示主流程。

系统监控数据可用于实时展示，也可按一定频率写入数据库，用于后续分析系统运行稳定性。

### 8.9 历史数据服务设计

历史数据服务负责为前端提供检测记录、车牌记录、拥堵统计、禁停事件、道路异常事件和系统运行记录查询能力。

历史查询应支持按时间范围、设备编号、任务类型和事件类型筛选。V1.0 阶段查询重点服务课程演示，不要求复杂权限、多租户或海量数据检索。

### 8.10 实时推送服务设计

实时推送服务基于 WebSocket 实现。前端建立连接后，云端向前端推送检测结果、车牌识别结果、拥堵热力图数据、禁停告警、道路异常告警、分析任务状态和系统资源状态。

WebSocket 消息应包含消息类型、时间戳、设备编号、任务编号和消息内容。当前端断开连接时，系统应允许前端重新连接，并通过 REST API 查询最近状态。

## 9. 边端采集模块设计

### 9.1 边端采集模块定位

边端采集模块负责在交通场景现场提供视频输入，并向云端上报设备状态。V1.0 阶段边端优先使用手机作为采集设备，可通过手机推流工具、手机浏览器页面或其他移动端采集方式完成视频采集。

边端不承担重型 AI 推理和历史数据管理职责。其核心职责是采集视频、传输视频、注册设备和上报状态。

### 9.3 视频采集设计

边端使用手机摄像头采集 704 智慧交通沙盘道路、车辆、闸机、禁停区域或异常区域画面。采集时应尽量保持摄像头固定、画面清晰、光照稳定，减少因画面抖动和遮挡造成的识别误差。

V1.0 阶段对边端视频采集能力的要求是满足课程演示，不要求开发完整原生手机 App。如果已有推流工具能够满足演示，可优先使用推流工具降低实现成本。

### 9.4 实时推流设计

实时推流用于体现系统实时感知能力。边端可以通过 RTSP/RTMP 等方式将视频流推送到云端，云端通过视频接入模块读取视频流并进行抽帧分析。

实时推流需要记录视频源地址、推流状态和关联设备编号。若推流失败或中断，边端或云端应能在设备管理和视频源状态中体现异常。

### 9.5 视频上传兜底设计

当实时推流条件不满足时，边端应支持上传短视频段作为兜底输入。上传视频进入云端后，应复用与实时流相同的分析任务、结果结构和前端展示方式。

视频上传不作为实时流的替代主方案，而是作为课程演示稳定性保障。上传视频应限制文件类型和文件大小，避免异常文件影响云端运行。

### 9.6 设备注册与心跳设计

边端启动后应向云端提交设备注册信息。注册信息包括设备名称、设备类型、设备备注和可选视频源信息。云端返回设备编号后，边端后续心跳和视频源状态均应携带该设备编号。

边端心跳用于上报在线状态、最近活动时间和视频源状态。V1.0 阶段心跳间隔可设置为 5-10 秒，云端根据心跳超时时间判断设备离线。

### 9.7 边端状态提示设计

边端应尽量向使用者展示基础状态，例如设备是否注册成功、云端连接是否正常、推流或上传是否成功。若边端采用现成推流工具，则状态提示可以主要由云端设备管理页面承担。

## 10. 前端展示模块设计

### 10.1 前端展示模块定位

前端展示端是系统的监控和管理入口，面向监控查看人员、平台管理员和模型配置人员。前端不直接执行视频推理或数据库写入，而是通过 REST API 和 WebSocket 与云端服务交互。

V1.0 阶段前端应优先服务课程演示和系统管理，不应设计成营销展示页。页面重点是实时画面、AI 结果、设备状态、模型配置、系统状态和历史数据。

图示说明：本图用于展示前端展示模块中的实时监控大屏、闸机监控、拥堵热力图、禁停区域监控、道路异常监控、设备管理、模型管理、系统状态和历史查询等页面关系。

### 10.3 实时监控大屏设计

实时监控大屏用于集中展示系统当前运行状态。页面应包含视频画面区域、检测结果叠加层、实时指标卡片、告警列表、任务状态和系统状态摘要。

视频画面区域展示边端采集的视频或上传视频分析画面。检测结果叠加层用于展示车辆框、车牌框、禁停区域和异常区域。指标卡片用于展示车辆数量、车牌识别数量、拥堵等级、告警数量和设备在线状态。

### 10.4 闸机监控视图设计

闸机监控视图用于展示车牌识别与白名单比对结果。页面应展示识别车牌号、车牌位置、识别置信度、白名单命中状态和模拟通行决策。

课程项目不要求真实控制闸机，因此前端只展示“允许通行”“未授权”或“需人工确认”等决策结果。

### 10.5 拥堵热力图视图设计

拥堵热力图视图用于展示道路区域车辆密度。页面应支持在视频画面或区域示意图上叠加热力图效果，并展示区域车辆数量、密度等级、拥堵等级和变化趋势。

热力图数据来自云端对车辆检测结果的区域统计和时间窗口统计。前端负责可视化展示，不负责直接计算拥堵业务逻辑。

### 10.6 禁停区域监控视图设计

禁停区域监控视图用于展示车辆在禁停区域内的停留情况。页面应展示禁停区域框、车辆跟踪 ID、车辆位置、进入时间、停留时长和超时告警状态。

当车辆停留时间超过阈值时，前端应突出显示告警信息，并在告警列表中记录事件类型、设备来源、发生时间和处理状态。

### 10.7 道路异常监控视图设计

道路异常监控视图用于展示疑似道路异常区域。页面应展示异常位置、异常区域、异常分数、影响车道、告警等级和发生时间。

道路异常检测结果来自云端 FastFlow 异常检测模型或规则检测兜底。前端应避免只展示抽象告警文字，应尽量在画面或区域图上标注异常位置。

### 10.8 设备管理页面设计

设备管理页面用于展示边端设备列表、在线状态、最近心跳时间、视频源状态和当前任务状态。平台管理员可通过该页面判断手机是否成功接入云端，视频源是否正常传输。

V1.0 阶段设备管理页面以展示和状态监控为主，不要求复杂设备权限管理。

### 10.9 模型管理页面设计

模型管理页面用于展示和配置模型运行参数。页面应支持查看模型名称、模型类型、运行状态、推理频率、置信度阈值、异常分数阈值、检测区域、禁停区域、拥堵阈值、停留时间阈值和模拟模式开关。

模型配置保存后由云端统一管理，并在分析任务启动或重启时生效。V1.0 阶段不要求在线训练模型。

### 10.10 系统状态页面设计

系统状态页面用于展示云端服务运行情况，包括 CPU 使用率、GPU 使用率、内存使用率、视频传输状态、拉流状态、分析任务状态和 WebSocket 推送状态。

当本地环境无法获取 GPU 信息时，前端应显示“未检测到 GPU”或“GPU 监控不可用”，并明确该状态不影响系统主流程演示。

### 10.11 历史查询与统计页面设计

历史查询与统计页面用于查询车牌记录、拥堵趋势、违规停车事件和道路异常事件。页面应支持按时间范围、设备编号、事件类型和任务类型筛选。

统计展示可包括检测车牌数量、白名单命中数量、拥堵等级变化趋势、违规停车数量、道路异常数量和异常位置分布。

## 11. 视觉感知算法模块设计

### 11.1 算法模块定位

视觉感知算法模块负责从视频帧中生成结构化交通感知结果。算法模块部署在云端，由分析任务服务调用，不直接与前端或边端耦合。

V1.0 阶段算法模块应同时支持真实模型推理和模拟结果兜底。真实模型用于展示项目技术能力，模拟结果兜底用于保证模型文件、硬件性能或演示环境不稳定时系统链路仍然可用。

### 11.2 视觉感知算法功能模块图

【图 11-1 视觉感知算法功能模块图：待绘制】

图示说明：本图用于展示算法模块中的视频帧输入、预处理、车辆检测、车牌识别、拥堵分析、禁停跟踪、道路异常检测、模拟结果兜底和结构化结果输出之间的关系。

### 11.3 算法输入设计

算法模块输入主要来自分析任务服务，包括视频帧、设备编号、任务编号、时间戳、模型配置、检测区域、禁停区域、拥堵阈值和模拟模式开关。

视频帧进入算法模块前，可进行尺寸缩放、颜色空间转换、画面裁剪、区域掩膜和图像增强等预处理。预处理应服务于模型稳定性和推理效率，不应改变原始业务语义。

### 11.4 车辆检测设计

车辆检测采用 YOLO 系列轻量检测模型。模型输入为视频帧，输出为车辆目标框、车辆类别、置信度和来源帧信息。

车辆检测结果是拥堵分析、禁停跟踪和部分车牌识别流程的基础。系统应保存主要车辆检测结果，并将可视化所需字段推送至前端。

### 11.5 车牌识别设计

车牌识别采用 YOLO 车牌定位 + PaddleOCR 字符识别 + 车牌规则校验的技术路线。系统首先定位车辆和车牌区域，然后对车牌区域进行裁剪、增强和校正，再调用 PaddleOCR 识别车牌字符。

识别结果应包含完整车牌号、数字字符、车牌框、OCR 置信度和白名单比对结果。若车牌区域过小、模糊、遮挡或 OCR 置信度不足，系统应将结果标记为“需人工确认”或在演示模式下使用预设样例结果。

V1.0 阶段不引入其他 OCR 主方案，正式技术路线以 YOLO + PaddleOCR 为准。

### 11.6 拥堵分析设计

拥堵分析采用 YOLO 车辆检测 + 区域密度规则 + 时间窗口统计的技术路线。系统根据车辆检测框中心点所属区域统计车辆数量，并结合区域容量计算车辆密度和拥堵等级。

拥堵等级可采用低、中、高三档，也可根据演示需要扩展为更细粒度等级。为降低单帧误检影响，系统应结合时间窗口统计，使热力图变化更加稳定。

### 11.7 禁停跟踪设计

禁停跟踪采用 YOLO 车辆检测 + ByteTrack 跟踪 + 禁停区域规则的技术路线。ByteTrack 根据连续帧检测框为车辆分配跟踪 ID，禁停区域规则判断车辆是否进入禁停区域并计算停留时间。

当同一车辆在禁停区域内停留超过预设阈值时，系统生成禁停超时告警。告警结果应包含车辆跟踪 ID、进入时间、停留时长、当前位置、告警等级和关联设备。

### 11.8 道路异常检测设计

道路异常检测采用 FastFlow 异常检测模型 + 规则检测兜底的技术路线。FastFlow 用于学习正常道路画面的特征分布，并在实时帧中发现疑似异常区域。

异常检测输出应包含异常区域、异常分数、影响车道、持续时间和告警等级。规则检测兜底可基于固定摄像头视角、道路区域掩膜、背景差分、静止区域变化、异常面积阈值和持续时间阈值判断疑似异常。

道路异常检测不把常规目标检测作为主方案或兜底方案，因为异常物体类型可能未知，难以提前穷举所有类别。

### 11.9 模拟结果兜底设计

模拟结果兜底用于在模型文件缺失、模型加载失败、识别效果不稳定或硬件性能不足时输出结构化演示数据。模拟结果应与真实模型输出保持相同数据结构，避免前端和数据库处理逻辑分裂。

模拟结果可基于预设样例车牌、固定车辆框、固定轨迹、区域拥堵数据和预设异常事件生成。模拟结果应明确作为演示兜底模式，不作为真实算法效果评估依据。

### 11.10 算法输出设计

算法输出应采用统一结构，至少包含时间戳、设备编号、任务编号、任务类型、目标类别、目标位置、置信度、跟踪编号、事件类型、区域编号和可视化字段。

不同算法能力可在统一结构基础上扩展字段。例如车牌识别扩展车牌号和白名单比对结果，拥堵分析扩展区域车辆数和拥堵等级，禁停跟踪扩展停留时长，异常检测扩展异常分数和影响车道。

## 12. 数据库与数据结构设计

### 12.1 数据设计目标

数据库设计用于支撑设备管理、视频源管理、分析任务管理、模型配置、白名单车辆库、检测结果、告警事件、统计分析和历史查询。

V1.0 阶段数据库设计应优先保证结构清晰、字段可理解、便于演示查询和后续扩展。系统不追求复杂分库分表或高并发优化。

### 12.2 数据库选型

V1.0 阶段建议采用 SQLite 作为默认数据库，降低本地开发和课程演示部署复杂度。若后续系统扩展到多设备持续接入、多任务并行分析和大量历史数据查询，可切换到 PostgreSQL。

数据库访问应通过数据访问模块封装，避免业务逻辑直接依赖具体数据库类型。

## 13. 接口与通信协议设计

### 13.1 接口设计目标

接口设计用于明确边端、云端和前端之间的数据交互方式。V1.0 阶段接口设计重点服务设备接入、视频上传、分析任务管理、模型配置、实时结果推送和历史查询。

系统采用 REST API、WebSocket 和视频流协议组合。REST API 用于请求响应式业务，WebSocket 用于实时结果推送，RTSP/RTMP 用于实时视频流传输，HTTP 文件上传用于视频段兜底输入。

### 13.2 REST API 设计原则

REST API 应遵循清晰、稳定、便于测试的原则。接口路径按资源类型划分，请求和响应采用 JSON 格式，上传视频文件时使用 Multipart 表单。

接口响应应包含状态码、消息说明和数据对象。发生错误时应返回明确错误信息，便于前端展示和联调排查。

### 13.3 设备管理接口

| 方法 | 路径 | 说明 | 主要参数 |
|---|---|---|---|
| POST | /api/devices/register | 注册边端设备 | name、type、remark |
| POST | /api/devices/{id}/heartbeat | 上报设备心跳 | status、video_source_status |
| GET | /api/devices | 查询设备列表 | status、keyword |
| GET | /api/devices/{id} | 查询设备详情 | id |

设备注册接口用于建立边端设备身份，心跳接口用于维护设备在线状态。设备列表接口用于前端设备管理页面展示设备状态。

### 13.4 视频源接口

| 方法 | 路径 | 说明 | 主要参数 |
|---|---|---|---|

实时视频流和上传视频文件应统一生成视频源记录，并进入后续分析任务流程。

分析任务接口用于连接视频源、模型配置和算法模块。任务状态变化应同步到前端实时监控页面。

| 方法 | 路径 | 说明 | 主要参数 |

### 13.9 WebSocket 消息设计

WebSocket 主要用于云端向前端推送实时消息。消息应包含消息类型、时间戳、设备编号、任务编号和数据内容。

| 消息类型 | 说明 | 主要内容 |
|---|---|---|
| detection_result | 检测结果 | 车辆框、车牌框、置信度、目标类别 |
| plate_result | 车牌识别结果 | 车牌号、白名单命中、通行决策 |
| congestion_update | 拥堵热力图更新 | 区域车辆数、密度等级、拥堵等级 |
| parking_alert | 禁停告警 | 跟踪 ID、停留时长、告警等级 |

| system_status | 系统状态 | CPU、GPU、内存、连接数 |

### 13.10 视频传输协议设计

实时视频流采用 RTSP/RTMP 作为主方案。边端将视频流推送至云端或视频服务，云端读取视频流并抽帧分析。

视频上传采用 HTTP Multipart 方式，云端保存视频文件路径和视频源记录，并将上传视频作为分析任务输入。

## 14. 部署与运行环境设计

### 14.1 部署目标

V1.0 阶段部署目标是支持课程演示环境下的本地运行。系统应能够在一台云端电脑、一台手机和一个浏览器前端之间完成端到端演示。

部署方案优先简单可靠，不强制使用容器编排、服务注册或复杂云平台。后续可根据项目进展补充 Docker 或云服务器部署方式。

### 14.2 运行角色

| 运行角色 | 设备 | 主要运行内容 |
|---|---|---|
| 边端采集设备 | 手机 | 视频采集、推流或上传、设备状态上报 |
| 云端分析服务器 | 电脑 | FastAPI 后端、视频接入、AI 推理、数据库 |
| 前端展示端 | 浏览器或另一台电脑 | 实时监控、管理配置、历史查询 |

### 14.3 开发环境要求

| 环境项 | 建议版本或说明 |
|---|---|
| Python | 3.10 及以上 |
| Node.js | 18 及以上 |
| 后端框架 | FastAPI、Uvicorn |
| 前端框架 | Vue 3、TypeScript、Vite |
| 数据库 | SQLite，后续可切换 PostgreSQL |
| 视频处理 | OpenCV、FFmpeg |
| 浏览器 | Chrome、Edge 或其他现代浏览器 |

### 14.4 运行环境要求

云端电脑应具备基本 CPU 和内存资源，能够运行后端服务、视频处理和前端开发服务。若具备 GPU，可用于模型推理加速；若没有 GPU，系统应支持 CPU 推理或模拟结果兜底。

手机和云端电脑应处于可互通网络环境中。若实时推流无法稳定运行，应使用短视频上传方式完成演示。

### 14.5 启动流程

系统启动流程建议如下：

1. 准备数据库、示例数据、白名单车辆库和模型配置。
2. 启动后端 FastAPI 服务。
3. 启动前端展示服务。
4. 手机端注册设备并开始推流或上传视频。
5. 云端创建分析任务并启动 AI 分析。
6. 前端打开实时监控页面，查看检测结果、告警事件和系统状态。
7. 前端打开历史查询页面，查看历史记录和统计数据。

### 14.6 数据与模型文件管理

大体积视频文件和模型权重不建议直接提交到代码仓库。项目应在 `data/` 和 `models/` 目录中保留说明文档、占位文件或下载说明。

白名单车辆库、测试样例和小规模历史数据可作为演示数据保存，便于系统启动后快速进入演示流程。

## 15. 安全性、可靠性与可扩展性设计

### 15.1 安全性设计

V1.0 阶段安全性设计以基础防护为主，不追求完整生产级权限体系。系统应对设备注册、视频上传、模型配置和历史查询等接口进行基本参数校验。

视频上传接口应限制文件类型和文件大小，避免无效文件或过大文件影响后端运行。模型配置接口应预留访问控制设计，避免未授权用户修改模型参数和区域配置。

### 15.2 可靠性设计

系统可靠性重点体现在异常可见、流程可恢复和演示可兜底。

| 异常场景 | 可靠性措施 |
|---|---|
| 设备离线 | 通过心跳超时标记离线并在前端展示 |
| 视频流中断 | 更新视频源状态，提示切换视频上传兜底 |
| 模型加载失败 | 记录错误信息，启用模拟结果兜底或提示模型不可用 |
| WebSocket 断开 | 前端尝试重连，并可通过 REST API 查询最近状态 |
| GPU 不可用 | 显示 GPU 监控不可用，不影响主流程 |
| 数据库写入失败 | 返回错误状态并记录异常信息 |

### 15.3 可扩展性设计

系统设计应支持后续扩展多设备、多模型、多场景和更完整的数据统计能力。

后端模块化单体结构为后续拆分服务保留边界。若后续需要扩展，可以优先将 AI 推理、视频处理和实时推送拆分为独立服务。

数据库设计保留设备、视频源、任务、模型配置、结果和事件之间的关系，便于后续从 SQLite 切换到 PostgreSQL。

### 15.4 可维护性设计

系统应保持清晰模块边界。后端按设备、视频、任务、模型、系统状态、历史查询和实时推送划分模块；前端按页面和业务视图划分模块；算法按车辆检测、车牌识别、拥堵分析、禁停跟踪和道路异常检测划分模块。

接口、数据结构和配置项应保持统一命名，减少前后端联调成本。

| UC-05 | 云端执行禁停跟踪 | 前端展示车辆停留时长和禁停超时告警 |
| UC-06 | 云端执行道路异常检测 | 前端展示异常区域、异常分数和道路异常告警 |
| UC-07 | 查询历史数据 | 历史查询页面显示车牌记录、拥堵统计和告警事件 |
| UC-08 | 模型或 GPU 不可用 | 系统显示可解释状态，并可启用模拟结果兜底 |

## 17. 风险分析与应对措施

### 17.1 技术风险

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 实时推流不稳定 | 视频无法进入云端分析流程 | 保留短视频上传作为兜底输入 |
| 模型加载失败 | 无法输出真实推理结果 | 启用模拟结果兜底并记录模型状态 |
| 硬件性能不足 | 分析延迟过高或帧率不足 | 降低抽帧频率、缩小分辨率、启用轻量模型 |
| 道路异常样本不足 | FastFlow 效果不稳定 | 使用规则检测兜底和预设演示样例 |
| 前后端联调复杂 | 页面无法及时展示结果 | 优先定义统一接口和 WebSocket 消息格式 |

### 17.2 演示风险

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 现场网络不稳定 | 手机与云端连接失败 | 提前准备上传视频样例 |
| 手机画面抖动 | 检测和识别效果下降 | 使用固定支架或固定拍摄角度 |
| 光照不足 | 车牌识别和异常检测效果下降 | 提前调整拍摄环境和示例视频 |
| 评审现场模型运行异常 | 演示中断 | 准备模拟结果模式和历史样例数据 |

### 17.3 数据风险

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 白名单数据不足 | 无法展示通行决策 | 准备固定测试车牌和白名单样例 |
| 历史数据为空 | 历史查询页面无内容 | 预置少量演示记录 |
| 视频样例不足 | 无法覆盖多类场景 | 提前录制车牌、拥堵、禁停和异常样例 |

### 17.4 进度风险

系统涉及边端、后端、前端、算法和数据多个模块，联调复杂度较高。V1.0 开发应优先保证最小闭环，再补充高级效果。

建议开发顺序为设备注册和视频输入、云端任务和模拟结果、前端实时展示、真实模型接入、历史查询和统计展示。

## 18. 后续迭代方向

### 18.1 需求分析 V2.0 迭代方向

需求分析 V2.0 可进一步细化用户角色、权限边界、异常处理、性能指标、演示场景和验收标准。

后续可根据实际开发情况调整 P0、P1、P2 优先级，使需求与实现进度保持一致。

### 18.2 系统设计 V2.0 迭代方向

系统设计 V2.0 可进一步细化接口字段、数据库 ER 图、前端页面原型、部署脚本、模型配置文件格式和算法性能指标。

若后续系统规模扩大，可考虑拆分视频处理服务、AI 推理服务和实时推送服务，并引入 PostgreSQL、Redis 或消息队列等组件。

### 18.3 系统实现迭代方向

系统实现建议按以下顺序推进：

1. 建立 `backend`、`frontend`、`edge`、`data`、`models`、`tests` 目录结构。
2. 实现设备注册、心跳和视频上传接口。
3. 实现分析任务和模拟结果兜底流程。
4. 实现前端实时监控大屏和设备管理页面。
5. 接入 YOLO、PaddleOCR、ByteTrack 和 FastFlow 等算法模块。
6. 实现模型管理、系统状态和历史查询页面。
7. 完成系统联调、测试报告和演示文档。

### 18.4 展示效果迭代方向

后续可增强实时监控大屏的可视化效果，包括检测框叠加、车牌结果标注、热力图半透明覆盖、告警闪烁提示、趋势图表和演示模式切换。

展示增强应服务于监控和管理场景，避免将前端做成与业务无关的装饰页面。

### 19.3 V1.0 设计结论

系统设计方案 V1.0 采用云边端协同的分层模块化架构。边端手机负责视频采集、推流或上传、设备注册和心跳上报；云端服务负责视频接入、AI 分析、业务处理、数据存储和实时推送；前端展示端负责实时监控、管理配置、系统状态和历史查询。

V1.0 阶段系统优先保证可演示闭环，并通过视频上传、模拟结果兜底和样例数据降低演示风险。后续可在此基础上逐步完善真实模型效果、前端交互、接口细节、部署脚本和测试覆盖。
