# 云边端协同的智慧交通视觉感知系统设计报告 V2.0

## 1. 引言

### 1.1 编写目的

本文档规定云边端协同的智慧交通视觉感知系统的架构、模块、数据、接口、部署、安全、可靠性和测试设计，明确移动 Web 边端、云端服务与前端展示端的职责边界以及实时任务、视频链路、视觉业务和平台管理的实现方式。

本文档面向系统开发人员、测试人员、部署人员、课程评审人员和后续维护人员。

### 1.2 设计依据

| 依据 | 设计用途 | 优先级 |
| --- | --- | --- |
| 需求分析报告 V2.0 | 确定业务范围、角色权限、状态、数据生命周期和验收标准 | 最高 |
| 704 智慧交通沙盘与既定硬件 | 约束真实数据采集、算法验证和部署方式 | 环境依据 |

### 1.3 系统范围

系统处理由手机摄像头实时采集并通过 WebRTC 发送的分析任务，不提供视频上传或其他非实时分析入口。实时业务闭环为：边端注册与待命、前端创建任务、云端原子占用全局任务名额、边端视频上行、云端留存与分析、云端视频下行、前端结果叠加、任务结束与完整性检查、历史追溯、告警处理和单任务报告归档。

系统包括五类场景模板：闸机通行监控、道路交通态势、禁停区域监控、道路异常监控和综合监控。视觉业务包括车牌识别与白名单比对、车辆识别与拥堵分析、禁停区域车辆跟踪、道路障碍与遗撒物检测。

智能分析报告是云端内部的低优先级业务模块，仅针对一个已结束且数据完整的任务生成固定事实和分析性正文。系统不建设聊天入口、自然语言检索入口，也不允许外部模型改变系统配置、任务状态、告警事实或其他业务数据。

### 1.4 关键设计约束

| 约束 | 设计结论 |
| --- | --- |
| 任务并发 | 全系统仅有一个活动任务名额 |
| 异常占位 | 异常任务在四类活动资源全部停止前继续占用名额 |
| 边端形态 | 手机浏览器中的移动 Web 页面，使用注册码注册 |
| 媒体 | 只采集视频，不申请麦克风权限，不传输音频 |
| 视频链路 | 边端至云端和云端至前端为两段独立 WebRTC 会话 |
| 标注 | 云端发送原始实时画面与结构化结果，前端负责叠加 |
| 留存 | 云端留存完整任务视频，MySQL 保存元数据与业务数据，本地磁盘保存文件 |
| 生命周期 | 完整任务视频和普通逐帧结果按需求规定精确保留 7 天 |
| 算法 | 当前仅规定候选路线，必须经真实数据验证后冻结 |
| 数据真实性 | 不以预置检测数据或替代性输出支撑业务验收 |
| 性能指标 | 不填写未经实测的数值，按规定流程测量并冻结 |

### 1.5 术语与统一编码

| 术语 | 含义 |
| --- | --- |
| 活动任务 | 状态为已创建、启动中、运行中、结束中，或活动资源尚未清理完的异常任务 |
| 活动资源 | 四项释放核验对象：边端采集、视频会话、算法进程、任务视频写入器；视频会话项必须分别确认上行与播放下行均已关闭 |
| 视频会话 | 云端为一次任务输入建立的统一媒体标识，关联上行、下行、帧和结果 |
| 普通逐帧结果 | 尚未提升为长期业务记录的帧级目标、区域状态和算法分数 |
| 配置快照 | 任务创建时固化的模板版本、功能、模型、区域、阈值、规则和白名单版本 |
| 数据版本 | 告警等并发更新对象的乐观锁版本号 |

数据库和接口内部使用稳定英文枚举，界面显示中文名称。任务状态依次编码为 `CREATED`、`STARTING`、`RUNNING`、`STOPPING`、`ENDED`、`ABNORMAL`；边端状态编码为 `OFFLINE`、`IDLE`、`EXECUTING`、`ABNORMAL`；告警状态编码为 `PENDING`、`PROCESSING`、`RESOLVED`；报告状态编码为 `GENERATING`、`DRAFT`、`PENDING_REVIEW`、`ARCHIVED`、`REJECTED`、`REVOKED`、`GENERATION_FAILED`。

## 2. 总体架构设计

### 2.1 架构风格与系统组成

系统采用由移动 Web 边端、云端服务和前端展示端组成的三端架构。边端负责现场采集与任务执行，云端负责视频、算法、业务、数据和智能分析，前端负责业务操作与结果展示。

云端采用模块化单体架构，接入、业务、算法、智能分析、数据和运行支撑模块统一部署在同一云端工程与云端电脑中，并通过明确的模块契约隔离职责。视频处理进程和算法工作进程由后端管理为独立操作系统进程，统一归属云端服务，共享任务编号、配置快照、状态规则和持久化边界。

云端智能分析层的大模型调用适配器通过 HTTPS API 调用外部多模态模型服务，其接口约定遵循第 8 章，角色、输入和数据边界遵循第 10 章。

### 2.2 总体架构分层

| 端 | 层次 | 主要内容 | 作用 |
| --- | --- | --- | --- |
| 移动 Web 边端 | 采集与控制层 | 移动 Web 页面、设备注册、摄像头权限、任务指令、状态上报、WebRTC 视频上行 | 负责现场真实视频采集、任务指令执行和边端状态反馈 |
| 云端服务 | 接入层 | HTTPS REST、边端控制 WSS、前端实时 WebSocket、WebRTC 信令、视频上行接收、视频播放下行、统一帧编号、算法取帧、任务视频留存和媒体会话清理 | 负责三端通信接入、实时视频处理、视频留存以及视频与分析结果关联 |
| 云端服务 | 业务层 | 用户认证、权限校验、设备管理、任务管理、场景模板、模型管理、白名单、业务规则、交通告警、历史查询和文件管理 | 负责系统业务流程、状态管理、配置管理、确定性规则和平台管理 |
| 云端服务 | 算法层 | 车辆与车牌检测、车牌字符识别、拥堵分析、车辆跟踪、道路异常检测和结构化结果标准化 | 负责从真实视频帧生成结构化视觉分析结果 |
| 云端服务 | 智能分析层 | 任务事实汇总、候选截图选择与复核、大模型调用适配器、分析正文校验、报告审核归档和 PDF 生成 | 负责基于单次已结束且数据完整的任务生成智能分析报告 |
| 云端服务 | 数据层 | MySQL、任务配置快照、逐帧结果、长期业务记录、完整任务视频、关键截图和报告 PDF | 负责结构化数据与文件的持久化、关联、查询和生命周期管理 |
| 云端服务 | 运行支撑层 | CPU、GPU、显存、内存和磁盘监控，服务健康、系统异常、操作审计、到期清理和文件一致性检查 | 负责系统运行监控、异常定位、数据清理和运维保障 |
| 前端展示端 | 展示与交互层 | 登录、统一实时监控、任务管理、设备管理、场景与分析配置、白名单、告警、历史查询、报告管理和系统管理 | 负责用户操作、实时视频播放、结构化结果叠加、平台管理和历史追溯 |

接入层包含业务协议接入和实时媒体处理两个模块组。业务协议接入模块组包含 REST API、边端控制 WSS、前端实时 WebSocket 和 WebRTC 信令，负责身份校验、请求解析、命令下发、状态上报、结果推送与信令交换；实时媒体处理模块组包含上行接收、下行播放、统一帧标识、算法取帧、有界帧队列、任务视频留存和媒体会话管理，负责媒体轨处理、留存及活动媒体资源清理。

### 2.3 核心协同链路

| 链路 | 起点与终点 | 内容 | 关联主键 |
| --- | --- | --- | --- |
| 边端控制 | 手机边端与云端 | 注册结果、心跳、启动/结束指令、执行回执 | `device_id`、`task_id`、`command_id` |
| 视频上行 | 手机边端到云端 | 无音频实时视频轨 | `task_id`、`device_id`、`video_session_id` |
| 视频下行 | 云端到前端 | 与任务输入同源的实时视频轨 | `task_id`、`source_id`、`video_session_id` |
| 分析链路 | 云端媒体到算法 | 有界队列中的采样帧 | `video_session_id`、`frame_seq`、`media_time_ms` |
| 结果链路 | 云端到前端 | 帧级结果、任务状态、告警和服务状态 | `task_id`、`video_session_id`、`event_id` |
| 留存链路 | 云端媒体到本地磁盘 | 完整任务视频及其文件元数据 | `task_id`、`file_id` |
| 管理链路 | 前端与云端 | 配置、任务、查询、处理和报告 REST 操作 | 用户身份与业务对象编号 |

## 4. 用户、设备与配置设计

| 在模板范围内调整任务参数 | 允许 | 允许 |
| 在模板允许的参考区域内调整本次任务区域 | 允许 | 允许 |

账号停用只阻止后续认证和接口访问，不删除该用户的任务、告警、审核或操作记录。任务控制、配置、白名单、告警、延期、异常干预和报告操作均写审计日志。两类角色均可通过系统状态接口查看 CPU、GPU、内存、磁盘和服务健康摘要；包含错误说明、关联对象和处置过程的系统异常详情，以及可追责的操作日志仅管理员可查，避免监控员接触诊断细节和审计数据。

### 4.2 移动 Web 边端

边端页面由云端通过受信 HTTPS 域名提供。首次使用时提交管理员创建的单次或受控有效注册码，云端校验后生成唯一 `device_id` 和设备凭证。注册码不得作为后续长期凭证；设备凭证存入浏览器安全存储并用于 WSS 和信令鉴权，管理员停用设备后凭证立即失效。

页面初始化顺序为：读取设备凭证、连接云端、显示摄像头权限、由用户完成权限授予、创建仅包含 `video: true` 且 `audio: false` 的媒体流、显示本地预览、建立控制 WSS、上报可执行状态。页面关闭、浏览器挂起、控制连接失效或权限撤销时，云端按心跳和连接状态将设备置为离线或异常。

| 边端模块 | 输入 | 输出 | 失败处理 |
| --- | --- | --- | --- |
| 注册模块 | 注册码、设备名称、浏览器信息 | 设备编号、设备凭证 | 显示注册码无效/过期/已使用 |
| 权限模块 | 用户授权 | 权限状态、本地视频轨 | 拒绝后保持不可选设备状态 |
| 控制模块 | WSS 指令 | ACK、执行阶段、心跳 | 重连并携带最近指令序号 |
| 采集模块 | 有效启动指令 | 无音频视频轨 | 摄像头失败上报任务相关错误 |
| 上行模块 | 视频轨、信令 | WebRTC 上行会话状态 | ICE/媒体失败上报并关闭会话 |
| 状态模块 | 页面与媒体状态 | 在线、权限、任务、指令结果 | 使用单调递增消息序号防止旧状态覆盖 |

边端仅执行与当前设备、当前任务一致且序号更新的指令。重复 `START_TASK` 或 `STOP_TASK` 返回相同最终结果，不重复创建媒体会话。任务结束后立即停止所有视频轨并关闭对等连接。

### 4.3 设备管理

设备在线判定同时使用有效控制连接和最近心跳。设备可用于创建任务必须同时满足：未停用、控制连接有效、心跳未过期、边端状态为空闲、摄像头权限有效、`current_task_id` 为空。判定窗口作为待冻结运行参数，通过断网和浏览器挂起测试确定。

管理员可创建设备注册码、修改名称和说明、查看注册时间、浏览器与系统信息、最近心跳、摄像头权限、当前任务及最近错误。设备编号一经生成不可修改；历史任务保留设备编号引用。

### 4.4 模型与场景模板

模型记录按“模型定义 + 不可变版本”管理。管理员可以登记、启用或停用模型版本，查询文件/服务标识、配置、适用功能、文件校验、加载状态和运行状态。已被任务快照引用的版本禁止物理删除。

场景模板采用可版本化结构。编辑已发布模板时创建新版本，不覆盖旧版本。每个版本保存默认功能、可选功能、依赖图、候选模型版本、归一化多边形区域、区域调整策略、参数默认值与允许范围、告警等级规则、前端面板配置和综合组合验证标记。区域调整策略逐区域声明 `adjustable`、允许编辑的几何类型、最少/最多顶点数、参考区域 `reference_region` 和允许包络 `allowed_envelope`；未声明可调整的区域只能使用模板默认几何。

任务创建时，管理员或监控员可在监控页的视频预览上拖动模板允许调整的检测区、禁停区、道路区或车道区顶点。前端提交归一化坐标，不提交像素坐标；后端以模板版本为准重新校验区域编号和类型、点数、坐标均在 `[0,1]`、多边形不自交且面积非零，并校验全部顶点位于该区域的 `allowed_envelope` 内且调整后区域与 `reference_region` 保持模板规定的包含或相交关系。越界返回字段级 `TASK_CONFIG_OUT_OF_RANGE`。

`POST /tasks` 的配置主体包含 `device_id`、`template_version_id`、`enabled_capabilities`、`parameter_overrides` 和可选 `region_overrides`；后者每项结构为 `{"region_id":"...","geometry":{"type":"POLYGON","points":[{"x":0.1,"y":0.2},...]}}`。后端校验模板和版本启用、设备可用、功能及依赖、综合组合、模型版本、参数范围、区域规则和白名单快照后，将模板默认值与合法覆盖合并为不可变任务配置快照，快照同时保留模板原区域、任务有效区域、覆盖内容和模板版本。任务覆盖只属于该任务，不更新或生成场景模板版本；任务事务提交后，包括启动中、运行中、结束中和异常阶段，均禁止修改功能、模型、区域和阈值。

### 4.5 白名单

管理员维护车牌号码、车辆说明、有效状态和更新时间。每次修改记录操作者。任务创建事务为启用车牌业务的任务生成白名单快照版本和快照明细；任务期间只按该版本比对，后续白名单修改不影响任务事实。车牌记录保存快照版本和比对时间。

## 5. 实时任务与媒体设计

### 5.1 全局任务互斥

MySQL 中建立唯一一行 `global_task_slot`，固定主键为 `1`，字段包含 `occupied_task_id`、`slot_version`、`occupied_at`。创建任务时开启事务并对该行执行排他锁；若 `occupied_task_id` 非空则返回冲突。校验设备与配置后，事务内创建任务及快照、将设备绑定任务并写入名额，提交后才下发启动指令。

`global_task_slot` 行锁对并发任务创建进行串行化，两个并发请求中仅一个可以占用名额并提交。创建前校验失败不产生任务；事务提交后的启动失败产生异常任务并继续占用名额。

### 5.2 任务状态机

| 当前状态 | 允许进入状态 | 进入条件 | 关键动作 |
| --- | --- | --- | --- |
| 无记录 | 已创建 | 名额、设备、配置在同一事务内校验成功 | 生成任务、快照、状态日志并占位 |
| 已创建 | 启动中 | 创建事务提交后立即编排 | 下发启动命令，记录启动阶段 |
| 启动中 | 运行中 | 上行、留存、下行准备和所选算法均确认可用 | 记录开始时间并推送状态 |
| 启动中 | 异常 | 持续阻断故障 | 记录异常并启动清理 |
| 运行中 | 结束中 | 管理员或监控员发出结束请求 | 阻止新输入，开始收尾 |
| 运行中 | 异常 | 持续阻断故障 | 记录异常并启动清理 |
| 结束中 | 已结束 | 文件封装、汇总、完整性检查和活动资源停止 | 设置结束时间并释放名额 |
| 结束中 | 异常 | 收尾出现阻断性故障 | 转异常清理/修复 |
| 异常 | 异常 | 清理重试或清理失败 | 更新清理阶段，不伪造新主状态 |

已结束为终态，不能恢复运行。异常任务不因写入 `ABNORMAL` 自动释放设备或名额。每次状态变化写入旧状态、新状态、阶段、触发来源、操作者/系统、时间、结果、错误代码和说明。

任务 `stage` 是六种固定主状态下的诊断字段。启动阶段依次记录为下发边端、等待上行、启动留存、准备下行、启动算法、运行确认；结束阶段依次记录为停止边端、停止取帧、排空处理中数据、关闭会话、停止写入与封装、汇总、完整性检查、释放资源，阶段值不扩展主状态集合。

### 5.3 正常启动与结束

创建事务提交后，任务编排器将状态更新为启动中并通过边端控制 WSS 下发含任务编号、指令编号和信令参数的启动命令。边端 ACK 后建立上行；云端创建统一视频会话、启动文件写入器、允许前端建立独立下行、启动所选算法工作器。全部必要组件就绪后进入运行中。

结束请求使用幂等键。首次有效请求把运行中改为结束中；重复请求返回当前状态。云端下发停止命令，停止向算法队列加入新帧，保存已接收结果，关闭上行和下行，停止算法，停止写入并封装视频，汇总统计和性能，执行完整性检查。资源确认停止后，事务内把任务改为已结束、设备改为空闲、清空全局名额。

### 5.4 异常清理与强制终止

短时网络波动、单帧失败或单次推送失败只记录运行异常和指标。阻断判定依据持续时间与连续失败次数，数值经真实环境测试冻结。阻断后记录异常环节、错误代码和说明，任务进入异常并启动幂等清理。

| 清理核验项 | 完成证据 |
| --- | --- |
| 边端采集停止 | 边端停止 ACK，或连接失效后云端确认无活动媒体接收 |
| 视频上行与播放下行会话关闭 | 上行对等连接终止且接收器无活动轨；该任务全部下行对等连接终止 |
| 算法进程停止 | 工作器退出/任务取消且队列不再消费 |
| 任务视频写入停止 | 写入器关闭且文件句柄释放 |

自动清理按当前阶段重试。只有上述项目全部确认，才在事务内释放设备和名额并记录 `resource_cleanup_completed_at`。管理员强制终止调用同一核验器，不能绕过停止确认。文件封装修复、数据汇总和文件一致性检查可在名额释放后进入数据修复队列，并保留异常和操作日志。

### 5.5 两段 WebRTC 与统一帧标识

视频上行与播放下行采用两个独立 WebRTC 会话。上行会话由边端发起，云端接收单路视频轨，为输入创建 `video_session_id` 和 `source_id`，对进入媒体管道的帧分配单调递增 `frame_seq`，以媒体时间 `media_time_ms` 作为业务时长基准并记录接收时间。

下行会话由前端针对运行任务单独协商。云端转发同源画面但不绘制标注。下行重连创建新的播放连接，但结构化结果仍必须携带统一输入 `video_session_id`；若输入会话因恢复而改变，云端明确推送 `video_session_changed`，前端清除旧结果。

算法取帧使用有界队列。队列满时按冻结后的策略丢弃旧待处理帧，优先保持接近实时，不允许历史帧无限积压。每个结果回传原任务、源、视频会话、帧序号、媒体时间、处理完成时间和源尺寸。

### 5.6 前端结果同步

前端只接受当前任务和当前视频会话的消息；`frame_seq` 小于等于已显示序号的结果视为乱序；处理完成时刻或媒体时间超过冻结的有效窗口时视为过期。有效结果替换同业务类型的上一结果，叠加层不改变视频元素尺寸。

任务切换、播放重连、输入视频会话变化、结果过期或任务离开运行中时立即清除标注。坐标以源视频宽高为基准，结合 `object-fit` 后的实际显示区域、缩放比例和留白偏移转换，支持窗口缩放和全屏。

### 5.7 任务数据完整性

| 检查类别 | 必检内容 | 通过标准 |
| --- | --- | --- |
| 公共项 | 状态日志、配置快照、视频封装与校验、算法运行、任务统计、性能汇总 | 每项存在且状态一致 |
| 功能项 | 启用功能的业务记录、统计、告警合并结果 | 有结果或明确合法零结果 |
| 关联项 | 任务、设备、视频会话、文件、截图、业务记录关联 | 外键/业务键有效 |

检查结果保存通过状态、时间、逐项结果和缺失项。检查失败不改变已结束主状态，允许查看已有数据，但报告生成接口拒绝请求。管理员重新汇总会创建新的检查记录，不覆盖原失败证据。

## 6. 视觉感知与业务规则设计

### 6.1 算法公共契约

算法输入包括任务配置快照、模型版本、归一化区域和统一编号视频帧；输出为标准结构化结果，不直接写告警状态。每个工作器写入启动、加载、帧处理、错误和停止运行记录。算法可在保持输入输出契约的前提下替换，但替换必须经过真实数据验证与版本冻结。

候选算法路线经真实数据验证后冻结最终选型、运行参数和量化指标。系统不得用人工预置结果代替真实视频推理结果完成验收，也不得以未经验证的阈值作为验收标准。

### 6.2 车牌识别与白名单比对

候选流水线为 YOLO 车辆/车牌检测、图像质量评估、PaddleOCR 字符识别、车牌格式校验和快照白名单比对。仅对满足质量和格式条件的完整字符执行白名单判定。

| 条件 | 输出 |
| --- | --- |
| 字符可靠且在任务白名单快照中 | 允许通行 |
| 字符可靠且不在快照中 | 未授权，并进入确定性告警规则 |
| 质量或字符可靠性不足 | 需人工确认，不判为未授权 |

长期车牌记录保存车辆框、车牌框、脱敏/原值受权字段、识别置信度、图像质量、判断、快照版本、比对时间、模型版本和关键截图。系统只展示通行判断，不输出物理闸机控制信号。

### 6.3 车辆识别与拥堵分析

候选流水线为 YOLO 车辆检测、区域归属、车辆计数、区域容量归一化、时间窗口平滑、拥堵等级判断和热力图数据生成。车辆密度、畅通/缓行/拥堵边界、时间窗口与持续阈值来自任务快照。

热力图数据与同一时刻区域统计使用相同车辆集合和区域定义，保存区域编号、车辆数、密度、等级、持续时间及媒体时间。拥堵条件持续满足后，由规则模块生成持续拥堵告警；状态下降时关闭持续事件的活跃更新，但不重新打开已处理告警。

### 6.4 禁停区域车辆跟踪

候选流水线为 YOLO 车辆检测、ByteTrack 跟踪、禁停区域判定、短时丢失关联和停留规则。停留时间使用视频媒体时间计算，不使用算法处理完成时间。

跟踪事件以任务、区域和稳定跟踪标识为键，保存轨迹、进入时间、最后出现时间、累计有效停留时间和证据。短时丢失容忍窗口由验证冻结；窗口内重新出现延续同一事件，超过窗口后结束该连续事件。同一车辆一次连续停留只创建一个可更新事件和一个禁停超时告警。

### 6.5 道路障碍与遗撒物检测

候选流水线为 FastFlow 异常区域检测、道路区域约束、前景连通区域、面积与持续规则、车道映射和干扰过滤。该能力面向道路障碍与遗撒物异常，输出异常多边形/包围框、异常分数、出现与持续时间、区域和受影响车道。

车辆正常经过、人员短时进入和光照变化通过持续时间、运动一致性、区域约束及验证后的规则过滤。只有位置、面积和持续条件同时满足时生成道路障碍告警。

### 6.6 确定性交通告警

外部模型不参与告警触发或分级。四类告警均由任务快照中的可测量条件产生，等级只能为低、中、高。

| 告警类型 | 触发事实 | 连续事件键 | 分级输入 |
| --- | --- | --- | --- |
| 未授权车辆 | 可靠车牌不在白名单快照 | 任务 + 车牌 + 业务窗口 | 识别质量、规则配置等已冻结字段 |
| 持续拥堵 | 区域拥堵条件持续达到阈值 | 任务 + 区域 + 活跃拥堵段 | 密度、持续时间、区域规则 |
| 禁停超时 | 连续停留超过阈值 | 任务 + 区域 + 跟踪事件 | 超时时长、区域规则 |
| 道路障碍 | 异常位置、面积、持续条件满足 | 任务 + 区域 + 异常事件 | 面积、持续时间、车道影响 |

规则引擎为事件生成稳定 `dedup_key`。同一连续事件采用插入或更新语义更新最近时间、持续时长、证据和等级，不按帧重复建告警。设备、媒体和算法故障进入系统异常，不计入交通告警。

### 6.7 告警状态与并发

告警严格按待处理、处理中、已处理迁移，不允许跳过处理中，不允许重新打开。新告警为待处理。领取事务执行带条件更新：仅当状态仍为待处理且版本等于请求版本时设置当前处理人、开始时间、状态和 `version + 1`；受影响行数为零返回冲突。

处理中告警仅当前处理人可按预期版本提交处理结果。管理员改派也必须校验预期版本，并保存原处理人、新处理人、原因和时间。每次变化在历史表保存前后状态、操作者、备注、时间和数据版本。

### 6.8 真实数据验证与冻结

四类场景分别按独立采集批次建立开发、训练/适配、验证和独立测试数据。每类包含正常、目标事件和干扰样本；验证集和测试集真值由人工确认，测试集不参与模型、参数或阈值选择。

| 阶段 | 产物 | 通过条件 |
| --- | --- | --- |
| 数据准备 | 批次清单、标注规范、人工真值、数据版本 | 数据用途隔离且可追溯 |
| 候选基线 | 模型版本、运行配置、指标计算程序 | 可在目标硬件完整运行 |
| 验证调优 | 验证集结果、误差分类、资源记录 | 完成候选比较和参数选择 |
| 设计冻结 | 最终路线、配置、阈值、组合清单、确认日期 | 评审确认并形成冻结记录 |
| 独立测试 | 测试集结果和系统测试报告 | 达到已冻结标准，测试期间不得下调 |

冻结指标包括需求列出的准确性、时效、帧率、结果有效时间、故障判定和资源阈值。记录必须包含硬件、系统版本、数据集版本、机位、输入条件、样本量、计算公式、原始结果、最终阈值和确认人。综合监控允许组合必须逐组合压测后登记。

## 7. 数据库与文件设计

### 7.1 存储原则

MySQL 8.4 LTS 保存业务事实、关系、状态、版本和文件元数据；云端本地磁盘保存完整任务视频、关键截图和报告 PDF。所有时间以带时区的统一服务时间保存，视频业务时长同时保留媒体时间。主键使用不可猜测的唯一标识，外键和业务唯一键共同保证关联。

逐帧结果与长期业务记录分表。算法原始结果不能被报告文本覆盖；报告保存事实快照和分析正文，两者字段隔离。客户端只能通过鉴权接口读取文件。

### 7.2 核心实体与表

| 表 | 关键字段 | 约束与用途 |
| --- | --- | --- |
| `users` | `id, username, password_hash, role, status, created_at, updated_at` | `username` 唯一；角色仅管理员/监控员 |
| `device_registration_codes` | `id, code_hash, status, expires_at, used_by_device_id, created_by` | 只保存摘要；状态和期限校验 |
| `devices` | `id, name, credential_hash, registration_code_id, registered_at, system_info, browser_info, description, status, camera_permission, last_heartbeat_at, current_task_id, enabled` | 设备编号唯一；保留注册来源与终端信息；当前任务受互斥约束 |
| `model_versions` | `id, model_code, name, version, capability, artifact_ref, config_json, enabled, file_status, load_status, run_status` | `(model_code, version)` 唯一；被引用版本保留 |
| `scene_template_versions` | `id, template_code, version, scene_type, config_json, enabled, published_at` | `(template_code, version)` 唯一且版本不可变 |
| `whitelist_vehicles` | `id, plate_number, description, enabled, updated_by, updated_at` | 规范化车牌索引 |

### 7.3 关系设计

用户创建任务并执行操作；设备可关联多个历史任务但同一时刻至多关联一个活动任务；模板版本和模型版本被多个任务快照引用；任务拥有视频会话、算法运行、帧结果、四类业务记录、告警、统计、性能、文件、完整性检查和报告版本。

一个交通告警属于一个任务，通过 `business_record_type + business_record_id` 关联产生该告警的长期业务记录，并可关联一个区域、一个关键截图和多条变更记录。业务服务校验被关联记录与告警的任务、业务类型和区域一致；业务记录删除受长期档案规则约束，不能留下告警孤立引用。禁停和道路异常历史按告警等级筛选时，使用该稳定关联连接对应交通告警。一个任务可有多个报告版本，但同时至多一个有效归档版本。文件通过 `related_type + related_id` 关联任务视频、截图或报告，并由业务服务额外校验对象所有权。

任务重新汇总时在同一事务中锁定该任务统计记录，创建递增 `version_no` 的新行，将旧行 `current_marker` 置为 `NULL` 并将新行置为 `1`；事务结束时必须存在且仅存在一个当前版本。历史查询和报告事实快照只读取该确定的当前行，同时保留任意数量的旧版本用于复核。

### 7.6 关键事务

| 事务 | 锁与条件 | 原子结果 |
| --- | --- | --- |
| 创建任务 | 锁全局名额行和设备行，校验配置 | 任务、快照、设备绑定、名额和状态日志同时提交 |
| 正常释放 | 锁任务、设备和名额，先验证资源清理证据 | 已结束、设备空闲和名额清空同时提交 |
| 异常释放 | 同上，要求 `cleanup_completed_at` 有效 | 异常任务保留，设备与名额同时释放 |
| 告警领取 | `status=PENDING AND version=expected` 条件更新 | 唯一处理人和变更日志 |
| 告警处理/改派 | 校验当前处理人/管理员、状态和版本 | 状态/处理人、版本和日志一致 |
| 延长保留期 | 锁任务相关视频文件和普通帧结果 | 两类到期时间同步更新并审计 |
| 报告正式版本替换 | 锁任务全部报告，校验新版本待审核 | 旧归档撤销、新版本归档和审核日志同时提交 |

数据库通过条件唯一约束或等效生成列约束保证每任务最多一个 `ARCHIVED` 报告；应用事务校验为第二层保护。替换失败必须整体回滚，不能留下两个归档版本或没有正式版本的中间提交。

### 7.7 本地文件目录

```text
storage/
  task-videos/{task_id}/{file_id}.mp4
  screenshots/{task_id}/{file_id}.{ext}
  reports/{task_id}/{report_id}/{file_id}.pdf
  quarantine/{check_batch_id}/
```

数据库仅保存相对路径。文件先写临时名，完成封装和校验后原子改名并把元数据状态由 `WRITING` 更新为 `AVAILABLE`。校验值、大小、格式、创建时间和状态必须与磁盘事实一致。隔离目录只用于一致性检查发现的孤立文件，不能被前端访问。

### 7.8 七天生命周期

完整任务视频和普通逐帧结果仅按以下起点计算生命周期：正常任务设置 `expires_at = end_at + 7 days`；异常任务从资源清理完成时间或管理员强制终止完成时间起设置精确 7 天到期时间。

管理员延长视频期限时，在一个事务中把该任务全部完整视频和普通逐帧结果更新为相同的新到期时间，并记录原因、操作者和旧/新期限。关键截图、车牌记录、拥堵时序、禁停事件、道路异常事件、交通告警、任务统计、性能、操作记录和报告不由常规到期任务删除。

定时清理器先领取到期批次，再删除文件或帧数据，最后记录完成结果。视频删除后保留 `media_files` 元数据、原到期时间和 `deleted_at`，不级联删除任务及长期记录。失败条目保持可重试状态并产生系统异常。

一致性检查双向执行：从 MySQL 检查可用文件是否存在且校验匹配；从受管目录检查是否存在无元数据文件。缺失文件记录为 `MISSING`，孤立文件进入隔离清单，均需审计后处理。

## 8. 接口与通信设计

### 8.1 通用 REST 约定

接口前缀为 `/api/v2`，请求和响应使用 UTF-8 JSON，文件流除外。写请求携带 `X-Request-Id`，任务创建和结束等幂等操作另携带 `Idempotency-Key`。分页使用 `page`、`page_size`，时间使用 ISO 8601 带时区格式。

```json
{
  "success": false,
  "code": "TASK_SLOT_OCCUPIED",
  "message": "全局任务名额已被占用",
  "request_id": "req_...",
  "details": {"active_task_id": "task_..."}
}
```

成功响应包含 `success=true`、`data` 和 `request_id`。后端不在错误详情返回密码摘要、设备凭证、本地路径、数据库语句或外部服务密钥。

### 8.3 边端控制 WSS

边端连接 `/ws/v2/edge-control?device_id=...`，握手时使用设备凭证。每条消息包含作为本次传输信封唯一标识的 `message_id`、`type`、`sent_at`、`device_id`；任务相关消息还包含 `task_id`。命令另含跨重发保持不变、用于幂等执行的 `command_id`，以及云端按设备严格单调递增、用于排序的 `command_seq`。重发命令使用新的 `message_id`，但保留原 `command_id` 和 `command_seq`；重连后边端上报最后处理序号，云端据此恢复未确认命令。

```json
{
  "message_id": "msg_...",
  "type": "START_TASK",
  "sent_at": "2026-07-10T10:00:00+08:00",
  "device_id": "dev_...",
  "task_id": "task_...",
  "command_id": "cmd_...",
  "command_seq": 18,
  "payload": {
    "video_session_id": "vs_...",
    "signaling_url": "wss://traffic.local/ws/v2/webrtc/uplink"
  }
}
```

对应 ACK 示例为：

```json
{
  "message_id": "msg_ack_...",
  "type": "COMMAND_ACK",
  "sent_at": "2026-07-10T10:00:01+08:00",
  "device_id": "dev_...",
  "task_id": "task_...",
  "command_id": "cmd_...",
  "command_seq": 18,
  "accepted": true,
  "execution_status": "COMPLETED"
}
```

边端发送 `EDGE_HELLO`、`HEARTBEAT`、`CAMERA_STATUS`、`COMMAND_ACK`、`UPLINK_STATUS`、`TASK_STATUS`；云端发送 `START_TASK`、`STOP_TASK`、`PING`。`COMMAND_ACK` 使用自己的 `message_id`，并必须回引 `command_id` 和 `command_seq`，另含 `accepted`、`execution_status` 和可选 `error_code`。边端以 `command_id` 缓存最终执行结果，以 `command_seq` 拒绝或回执过期命令；重复命令不得重复启动或停止资源。旧任务指令返回拒绝，不改变当前媒体状态。

### 8.4 前端实时 WebSocket

前端连接 `/ws/v2/realtime` 并使用用户访问凭证，订阅当前任务。消息公共信封如下：

```json
{
  "event_id": "evt_...",
  "type": "ANALYSIS_RESULT",
  "task_id": "task_...",
  "video_session_id": "vs_...",
  "source_id": "source_...",
  "frame_seq": 1258,
  "media_time_ms": 41933,
  "processed_at": "2026-07-10T10:00:42.125+08:00",
  "source_size": {"width": 1920, "height": 1080},
  "payload": {"business_type": "PLATE", "objects": []}
}
```

消息类型包括 `TASK_STATUS_CHANGED`、`TASK_STAGE_CHANGED`、`VIDEO_SESSION_CHANGED`、`ANALYSIS_RESULT`、`ALERT_CREATED`、`ALERT_UPDATED` 和 `SERVICE_STATUS_CHANGED`。WebSocket 提供连接期间的增量事件；客户端重连后先读取任务 REST 快照，再恢复增量订阅。

### 8.5 WebRTC 信令契约

上行信令路径 `/ws/v2/webrtc/uplink`，主体身份为设备；下行信令路径 `/ws/v2/webrtc/downlink`，主体身份为用户。两者交换 `offer`、`answer`、`ice_candidate`、`connection_state`，但会话和权限分别管理。

| 字段 | 上行要求 | 下行要求 |
| --- | --- | --- |
| `task_id` | 必须等于设备当前任务 | 必须等于用户订阅任务 |
| `video_session_id` | 由云端预分配 | 指向当前输入视频会话 |
| `device_id/source_id` | 与任务快照一致 | 与任务视频源一致 |
| `peer_id` | 注册设备编号 | 用户会话/浏览器实例编号 |
| `sdp_type/sdp` | 仅允许当前协商轮次 | 仅允许当前协商轮次 |
| `candidate` | 校验会话归属 | 校验播放权限与会话归属 |

媒体协商明确只接受视频轨。任何音频轨均被拒绝并记录协议错误。信令消息携带 `negotiation_id`，旧协商消息不得覆盖新连接。

### 8.6 错误码与 HTTP 状态

| 错误码 | HTTP | 含义 |
| --- | --- | --- |
| `AUTH_INVALID_CREDENTIALS` | 401 | 登录信息无效 |
| `AUTH_FORBIDDEN` | 403 | 角色或对象权限不足 |
| `DEVICE_CODE_INVALID` | 400 | 注册码无效、过期或不可用 |
| `DEVICE_NOT_READY` | 409 | 设备不在线空闲或权限无效 |
| `TASK_SLOT_OCCUPIED` | 409 | 全局名额被活动任务占用 |
| `TASK_STATE_CONFLICT` | 409 | 当前任务状态不允许操作 |
| `TASK_CONFIG_OUT_OF_RANGE` | 422 | 功能、依赖、区域或参数越界 |
| `TASK_RESOURCES_ACTIVE` | 409 | 强制终止前仍存在活动资源 |
| `VERSION_CONFLICT` | 409 | 乐观锁预期版本不匹配 |
| `MEDIA_NEGOTIATION_FAILED` | 502 | WebRTC 协商失败 |
| `FILE_NOT_AVAILABLE` | 404/409/410 | 文件不存在、尚未完成封装校验、已删除或已到期 |
| `INSUFFICIENT_STORAGE` | 507 | 空间无法保证完整留存 |
| `TASK_INCOMPLETE` | 422 | 任务完整性检查未通过 |
| `REPORT_STATE_CONFLICT` | 409 | 报告状态不允许操作 |
| `EXTERNAL_MODEL_FAILED` | 502 | 报告正文外部调用失败 |

### 8.7 幂等与并发规则

相同用户、接口和 `Idempotency-Key` 在有效窗口内返回首次提交的业务结果；`global_task_slot` 行锁独立负责不同请求键下的并发任务名额裁决。告警和报告状态操作必须同时校验状态、角色和版本。

### 9.1 页面结构

| 页面 | 核心区域 | 关键交互与状态 |
| --- | --- | --- |
| 登录 | 账号、密码、错误提示 | 登录成功按角色加载菜单 |
| 实时监控 | 设备/模板/功能/参数、视频、标注、任务状态、告警、专题面板 | 创建、结束、参数越界提示、重连与清标注 |
| 任务管理 | 列表、状态日志、阶段、异常、快照、完整性、关联数据 | 管理员重新汇总和强制终止 |
| 设备管理 | 注册码、设备列表、权限、心跳、当前任务 | 管理员维护，双方查看状态 |
| 场景与分析配置 | 模板版本、模型、功能依赖、区域、参数范围、等级规则 | 管理员发布新版本 |
| 白名单管理 | 车辆列表、状态、修改记录 | 管理员维护和筛选 |
| 告警管理 | 类型/等级/状态筛选、详情、证据、处理历史 | 领取、处理、管理员改派、冲突刷新 |
| 历史查询 | 任务、车牌、拥堵、禁停、道路异常、告警、性能、视频 | 多条件查询、保留期内回放 |
| 报告管理 | 截图复核、版本、事实、正文、审核、归档、PDF | 按角色显示操作 |
| 系统管理 | CPU/GPU/内存/显存/磁盘、服务、异常、用户、审计、清理 | 管理员操作，状态可查询 |

### 9.2 统一实时监控页面

页面保持任务状态、视频和当前告警始终可见。任务未开始时，左侧/顶部配置区选择在线空闲设备、模板、允许的功能和参数；后端校验失败时按字段显示越界原因。运行后配置区只读，主视频区播放云端下行，独立 Canvas 或 DOM 标注层覆盖其上。

专题面板按实际启用功能显示：闸机面板展示车牌、置信度、质量、白名单状态和通行判断；道路态势面板展示区域车辆数、密度、等级、持续时间和热力数据；禁停面板展示跟踪编号、轨迹、进入时间、停留时长和告警；道路异常面板展示异常区域、分数、持续时间、车道和等级。综合场景使用可切换面板，不重复创建视频播放器。

页面对 `TASK_SLOT_OCCUPIED` 显示当前占位任务；对 `VERSION_CONFLICT` 自动刷新告警详情并提示数据已变化；WebSocket 断开时保留视频连接状态但清除可能过期的标注，重连完成后读取 REST 快照。

### 9.3 区域与标注编辑

管理员在模板版本中以归一化坐标绘制检测区域、禁停区域、道路区域和车道多边形，并逐区域配置是否允许任务覆盖、几何约束、参考区域和允许包络。模板编辑器验证点数、坐标范围、自交、面积和必要区域存在性；发布后区域策略随模板版本不可变。

管理员和监控员在实时监控页创建任务时，可直接在当前设备视频预览上编辑模板标记为可调整的区域；不可调整区域显示为只读。前端即时显示参考区域和允许包络并执行预校验，提交 `region_overrides` 后以后端校验为准。合法有效区域连同模板原区域和覆盖差异写入任务快照，不回写模板；任务创建事务提交后编辑器锁定，运行期间不存在区域修改接口。

标注层按源尺寸换算车辆框、车牌框、轨迹、多边形和热力数据。绘制组件以业务类型隔离样式，消息过期时只清除对应业务层；会话变化时清除全部业务层。

### 9.4 历史与回放

历史页分别调用任务、车牌、拥堵、禁停、道路异常和告警端点，提供与各端点一致的时间、任务、设备及业务筛选、分页和排序，并从响应关联摘要进入任务详情、告警、截图、性能和报告。详情聚合配置快照、状态、长期业务记录、当前统计版本、性能、完整性、文件和报告，不依赖普通逐帧结果。

历史完整视频仅在文件为 `AVAILABLE`、未到期且未删除时使用受控 HTTP 视频端点播放。播放器通过 Range 请求加载并在用户拖动进度条时请求目标字节区间，不创建 WebRTC 会话；文件封装中显示不可播放状态，到期删除后明确显示到期时间和删除时间，仍可查看长期业务记录、截图和报告。

### 9.5 可用性与兼容性

移动边端以选定手机浏览器实测摄像头授权、前后台切换和 WebRTC；桌面前端以选定浏览器实测视频播放、WebSocket 和标注。桌面管理页面优先适配监控电脑，移动边端页面适配竖屏和横屏，所有关键状态使用文本与颜色共同表达。

## 10. 智能分析报告设计

### 10.1 边界与输入

报告以单个已结束且最新完整性检查通过的任务作为唯一输入，生成请求创建新版本并进入生成中。报告输入不包含历史任务范围查询、实时任务、跨任务汇总或系统操作。

云端首先固化事实快照，包括任务概况、配置、四类业务统计、告警与处理、关键事件、截图清单和性能汇总。事实字段由数据库查询和固定模板生成，外部模型输出不能写入这些字段。

### 10.2 截图选择与最小必要上下文

候选截图按启用的事件类型、等级、时间分布和去重规则选择。报告处于生成中时以 `WAITING_SCREENSHOT_REVIEW` 记录内部处理阶段，监控员可以取消不合适截图并确认最终清单；截图确认后才进入外部正文生成阶段，不增加报告主状态。外发数据只包含内部设备编号、脱敏车牌、任务统计、事件摘要、性能摘要和复核后的截图。

完整视频、用户信息、登录凭证、设备凭证、控制消息、本地文件路径、无关历史任务和未复核截图不得外发。外发请求摘要和服务标识进入报告审计，但密钥不落库。

### 10.3 生成与状态

外部模型只生成任务摘要、交通状态分析、重点事件分析、趋势解释、综合结论和处置建议。云端对返回内容执行章节结构和数据引用检查，再与事实模板组合；成功进入草稿，失败进入生成失败并保存错误阶段、代码和说明。

| 当前状态 | 允许进入状态 | 操作者/条件 |
| --- | --- | --- |
| 生成中 | 草稿、生成失败 | 云端生成成功或失败 |
| 草稿 | 待审核 | 生成该版本的监控员提交 |
| 待审核 | 已归档、已驳回 | 管理员归档或填写意见驳回 |
| 已归档 | 已撤销 | 管理员主动撤销或正式版本替换 |
| 已驳回 | 无 | 保留意见；重生成创建新版本 |
| 已撤销 | 无 | 保留内容、PDF 和记录 |
| 生成失败 | 无 | 重试创建新版本，不复用失败版本 |

### 10.4 正式版本替换与 PDF

归档前校验新版本为待审核、任务一致、事实快照完整且审核者为管理员。事务锁定任务报告集合，把现有有效归档版本改为已撤销并记录替换原因，再把新版本改为已归档；两步和审核记录一次提交。任何失败整体回滚。

仅当前有效归档报告可以新导出 PDF，PDF 由云端固定模板渲染，页面和 PDF 使用同一报告数据对象。已撤销版本已有 PDF 继续受控保留用于审计，但不作为有效报告对外导出入口。

## 11. 部署与运行设计

### 11.1 物理部署

云端电脑使用 Windows 11、AMD Ryzen 9 7945HX、32 GB 内存、NVIDIA GeForce RTX 4060 8 GB 显存和约 400 GB 可用 SSD。云端部署反向代理/证书入口、FastAPI 模块化后端、媒体处理、算法工作器、MySQL 8.4 LTS 和本地 `storage/`。

前端静态应用部署在独立电脑或由受信站点提供，通过桌面浏览器访问。手机、云端和前端电脑位于可互通局域网，使用固定局域网域名和受信证书。WebRTC 所需 UDP/TCP 媒体端口范围在部署配置中显式开放，数据库端口只绑定云端本机或受限管理网段。

### 11.2 配置与密钥

环境配置包括数据库连接、公开域名、证书路径、媒体端口范围、文件根目录、外部模型端点和密钥引用。生产演示配置不提交仓库；密钥通过环境变量或系统凭据存储加载。模板业务参数保存在数据库版本中，运行基础设施参数保存在部署配置中，二者不得混用。

### 11.3 启动与停止顺序

启动顺序为 MySQL、文件目录与权限检查、后端迁移和核心服务、媒体组件、算法模型预检、定时任务、HTTPS 入口、前端。系统只有在数据库、文件写入和任务核心模块健康时才允许创建任务；外部模型不可用只影响低优先级报告生成。

停止时先禁止新任务，若存在活动任务则执行正常结束或管理员确认的异常处置，然后停止实时连接、算法工作器、定时任务、后端和数据库。不得通过关闭进程绕过任务资源清理记录。

### 11.4 磁盘容量保护

任务创建前根据可用空间、留存写入能力和已知任务配置执行准入检查，任务时长、码率和空间安全余量阈值经目标环境实测后冻结。空间低于冻结准入阈值或无法保证完整留存时返回 `INSUFFICIENT_STORAGE`，不创建任务。运行中持续监控空间并产生系统异常，优先完成当前文件封装和受控结束。

## 12. 安全、可靠性与运维设计

### 12.1 安全控制

| 风险面 | 控制 |
| --- | --- |
| 用户身份 | 密码安全哈希、访问凭证有效期、停用实时校验、角色授权 |
| 设备身份 | 注册码摘要、受控有效期、独立设备凭证、停用撤销 |
| 传输 | 受信 HTTPS/WSS、WebRTC 加密媒体、信令鉴权 |
| 文件 | 仅相对路径入库、对象级鉴权、禁止目录遍历、受控流式读取 |
| 配置 | 后端范围校验、模板版本、任务快照、关键操作审计 |
| 并发 | 数据库事务、排他锁、乐观版本、幂等键 |
| 外部模型 | 最小必要、车牌脱敏、单任务、禁止完整视频和凭证外发 |
| 日志 | 敏感字段脱敏，不记录密码、凭证、密钥和完整 SDP |

### 12.2 故障处理

| 故障 | 判定与影响 | 处理 |
| --- | --- | --- |
| 边端断线 | 心跳和控制连接失效 | 短时等待重连；持续阻断则任务异常并清理 |
| 上行中断 | 无媒体帧或连接失败 | 记录重连；超过冻结条件转异常 |
| 下行中断 | 单个前端无法播放 | 重建下行，不影响上行和留存；无可用监控且构成阻断时按规则处理 |
| 单帧算法失败 | 个别帧处理错误 | 丢弃该帧并计数，不直接终止任务 |
| 算法持续失败 | 所选核心能力不可运行 | 记录阻断故障并清理任务 |
| 数据库失败 | 状态/结果不可可靠保存 | 停止接受新任务；活动任务受控结束或异常处理 |
| 文件写入失败 | 完整视频无法保证 | 记录系统异常并使任务进入阻断处置 |
| 空间不足 | 准入或运行检查失败 | 创建前拒绝；运行中受控结束 |
| WebSocket 中断 | 前端结果暂不可见 | 重连后读快照；清除过期标注 |
| 外部模型失败 | 报告正文无法生成 | 报告生成失败，可由监控员新建版本重试 |

### 12.3 监控与日志

系统管理采集 CPU、内存、GPU、显存和磁盘状态，并展示设备控制、WebRTC、算法、WebSocket、数据库、文件和外部模型调用状态。资源阈值经目标环境实测冻结。

日志按请求、任务、视频会话、设备、告警和报告关联。业务日志、系统异常和操作审计分开保存；任务状态及告警/报告操作以数据库记录为审计事实，文本日志用于诊断。系统时间保持同步，避免三端事件排序失真。

### 12.4 备份与恢复

MySQL 使用一致性备份，`storage/` 的长期文件与数据库文件元数据按同一备份批次标识。备份周期、保留份数和恢复目标经部署环境测量与恢复演练后冻结；课程验收前至少执行一次数据库与长期文件联合恢复演练，验证任务、事件、截图和报告关联。

常规 7 天到期数据无需作为长期恢复目标，但备份恢复不能使已删除视频重新成为可用文件而缺少元数据状态修正。恢复后先运行双向文件一致性检查，再开放文件访问。

### 12.5 可维护性

状态枚举、消息 Schema、错误码和数据库迁移纳入版本管理。算法适配器只输出统一领域对象，页面不依赖具体模型字段。每个模块提供健康检查和结构化错误，避免通过解析日志驱动业务状态。

## 13. 测试与验收设计

### 13.1 测试层次

| 层次 | 重点 | 方法 |
| --- | --- | --- |
| 单元测试 | 状态迁移、参数范围、坐标换算、规则合并、脱敏 | 固定输入断言领域输出 |
