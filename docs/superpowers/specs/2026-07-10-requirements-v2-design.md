# 云边端协同智慧交通视觉感知系统 V2.0 协作规格

## 1. 规格目的

本规格记录需求分析报告 V2.0 和系统设计报告 V2.0 的当前编写基线。后续文档应以本规格为依据，保持业务范围、任务流程、功能边界、数据关系、验收口径和技术约束一致。需要通过实践确定的算法选型与量化阈值，在验证完成后补充并冻结，不影响当前已确认业务范围。

本规格采用项目建设者视角描述已确认内容，不记录讨论过程，不描述未纳入 V2.0 的设想，不使用替代真实功能的模拟数据或预置结果。

## 2. 建设目标与范围

系统由移动 Web 边端、云端服务和前端展示端组成，面向学院 704 智慧交通沙盘完成真实视频采集、实时传输、视觉分析、结果展示、平台管理、历史追溯和智能分析报告生成。

V2.0 只建设实时分析任务，不建设视频文件上传和离线分析任务。单任务限制为系统全局互斥：同一时刻只允许一个任务处于已创建、启动中、运行中或结束中。异常任务在边端采集、两段视频会话、算法进程和任务视频写入完成停止前继续占用全局任务名额。正常任务进入已结束时释放设备与全局任务名额。

系统采用五类场景模板：闸机通行监控、道路交通态势、禁停区域监控、道路异常监控和综合监控。任务创建时先选择模板，再在模板约束范围内调整视觉功能、区域和阈值。

所有写入 V2.0 范围的需求均需实现并验收。开发优先级只表示实施顺序，不表示某项需求可以不实现。

## 3. 三端职责

### 3.1 移动 Web 边端

移动 Web 边端由云端电脑通过 HTTPS 提供页面。手机用户打开页面、完成设备注册并授予摄像头权限后，边端保持在线等待任务。

边端负责设备注册、任务指令接收、摄像头采集、WebRTC 视频上行和运行状态上报。边端只采集视频，不申请、传输或保存音频。

云端自动启动采集的前提是边端页面已打开、设备在线空闲且摄像头权限已授予。系统不要求在页面关闭、设备离线或权限未授予时唤醒摄像头。

### 3.2 云端服务

云端负责用户认证、设备控制、任务管理、WebRTC 视频上行与播放下行会话、任务视频留存、视频帧调度、视觉分析、业务规则、交通告警、MySQL 持久化、文件管理、历史查询、系统监控、智能分析报告和 PDF 生成。智能分析报告模块部署在云端内部，通过 HTTPS API 调用外部多模态大模型；外部大模型不作为独立系统端。

云端统一生成任务编号并管理任务状态。前端和边端均围绕云端任务编号进行交互。

### 3.3 前端展示端

前端部署在独立电脑上，负责登录、任务控制、实时监控、结构化结果叠加、设备与配置管理、告警处理、历史查询和报告管理。

前端通过 WebRTC 播放云端提供的实时视频，通过 WebSocket 接收任务状态、结构化分析结果和交通告警。检测标注由前端绘制，云端不生成带标注的视频流。

## 4. 部署与通信

云端电脑部署于 Windows 11，硬件基线为 AMD Ryzen 9 7945HX、32 GB 内存、NVIDIA GeForce RTX 4060 8 GB 显存和约 400 GB 可用 SSD 空间。业务后端、WebRTC 视频服务、视觉算法、MySQL 和本地文件存储均部署在该电脑上。

前端部署在另一台电脑上。手机、云端电脑和前端电脑处于同一可互通局域网。

系统采用局域网固定域名和受信证书。移动 Web 页面、前端页面、REST API、WebSocket 和 WebRTC 信令均通过安全连接访问。

边端到云端采用 WebRTC 视频上行，云端到前端采用独立的 WebRTC 播放下行。云端到前端采用 WebSocket 推送结构化分析结果。REST API 用于登录、配置、任务控制、查询、告警处理和报告操作。

## 5. 实时任务流程

任务启动流程如下：

1. 边端完成注册、摄像头授权和控制连接，进入在线空闲状态。
2. 监控员或管理员在前端选择边端设备、已有场景模板和模板允许的视觉功能组合，并在管理员设置的范围内调整本次任务参数。日常任务操作由监控员承担。
3. 前端调用云端接口创建并启动任务。
4. 云端在数据库一致性操作中完成全局任务名额校验与占用，生成任务编号并保存任务配置快照。
5. 云端立即向边端下发启动指令。
6. 边端启动摄像头并建立 WebRTC 上行会话。
7. 云端接收视频，准备前端 WebRTC 播放会话，启动任务视频留存和已选择的视觉功能。
8. 云端确认视频接入和分析流程正常后将任务置为运行中。
9. 前端播放实时视频，并叠加云端推送的最新有效结构化结果。

任务结束流程如下：

1. 监控员或管理员在前端提交结束任务请求。日常任务结束操作由监控员承担。
2. 云端将任务置为结束中，并向边端发送结束指令。
3. 边端停止摄像头采集和 WebRTC 上行会话。
4. 云端停止获取新的算法输入帧，完成已进入处理流程的数据保存。
5. 云端关闭任务视频会话，完成视频文件封装、任务数据汇总和完整性检查。
6. 云端完成任务收尾后将任务置为已结束。
7. 已结束任务可以进入历史查询；只有通过完整性检查的任务可以进入智能分析报告流程。

## 6. 任务与设备状态

任务主状态包括已创建、启动中、运行中、结束中、已结束和异常。正常迁移路径为“已创建 -> 启动中 -> 运行中 -> 结束中 -> 已结束”。启动中、运行中和结束中发生阻断性故障时可以进入异常。已创建是创建并启动操作中的短暂状态，不提供长期停留或人工启动入口。

任务主状态只表示生命周期。启动中和结束中可以记录当前处理阶段；任务发生阻断性故障时记录异常环节、错误代码、错误说明和发生时间，不为视频、播放和算法建立多组任务子状态。异常任务通过当前阶段记录自动清理、清理重试和清理失败；完成边端采集停止、视频上行与播放下行会话关闭、算法进程停止和任务视频写入停止后释放设备与全局任务名额。管理员强制终止时也必须先确认这些活动资源已经停止，未完成的任务视频文件封装修复、数据汇总和文件一致性检查转入后续数据修复流程。

移动 Web 边端状态包括离线、空闲、任务执行中和异常。

短时网络波动、单帧处理失败和单次结果推送失败不直接使任务进入异常。只有故障持续存在并导致任务无法继续完成实时监控或视觉分析时，任务才进入异常。具体持续时间和连续失败次数在端到端实践后冻结。

已结束任务的数据完整性由独立检查结果表示。检查项包括任务起止时间、配置快照、视频文件封装状态、已启用功能的结果汇总、告警合并、关键截图关联和性能汇总。检查失败时记录缺失项并允许管理员重新执行汇总；未通过完整性检查的任务可以查看已有数据，但不能生成报告。

## 7. 机位与场景约束

移动 Web 边端可以在不同任务之间调整拍摄位置。单次任务开始前必须完成手机固定、画面预览、模板选择和区域确认，任务运行期间使用支架或其他方式保持机位稳定。

闸机通行、道路交通态势、禁停区域和道路异常可以使用不同机位。综合监控只启用当前机位具备有效观测条件的视觉功能。

场景模板定义默认视觉功能、可选功能、模型配置、区域配置、业务参数默认值与可调整范围、告警分级规则和前端展示内容。监控员创建任务时可以在模板允许范围内选择视觉功能和调整本次任务参数，云端拒绝越界参数。任务创建时保存模板与配置快照，任务参数不反向修改模板，模板后续修改不影响运行中或已结束任务。

## 8. 视觉业务范围

### 8.1 车牌识别与白名单比对

系统检测车辆和车牌，识别满足图像质量要求的车牌字符，并与有效白名单比对。结果包括车辆框、车牌框、车牌号码、识别置信度、图像质量状态和白名单比对结果。

无法可靠识别的车牌标记为需人工确认，不直接判定为未授权车辆。

### 8.2 车辆识别与拥堵分析

系统基于车辆检测结果统计道路区域车辆数量和密度，通过区域容量与时间窗口判断畅通、缓行和拥堵状态，并生成热力图数据。

### 8.3 禁停区域车辆跟踪

系统对进入禁停区域的车辆进行连续跟踪，记录跟踪编号、轨迹、进入时间和停留时长。车辆持续停留超过配置阈值时形成禁停超时告警。

### 8.4 道路障碍与遗撒物检测

系统检测道路区域中新出现并持续存在、可能影响通行的障碍或遗撒物，输出异常区域、异常分数、出现时间、持续时间、受影响车道和告警等级。系统不以识别所有异常物体具体类别作为验收目标。

## 9. 候选技术路线

需求分析报告不固定具体算法。系统设计报告记录以下候选路线，并在真实数据和确定硬件环境中验证后冻结最终选型：

| 业务场景 | 候选技术路线 | 输出结果 | 当前状态 |
| --- | --- | --- | --- |
| 车牌识别与白名单比对 | YOLO 车辆及车牌检测 + PaddleOCR 字符识别 + 车牌格式校验 | 车辆框、车牌框、车牌号码、OCR 置信度、图像质量状态、白名单比对结果 | 待真实数据验证 |
| 车辆识别与拥堵分析 | YOLO 车辆检测 + 区域车辆统计 + 区域容量规则 + 时间窗口平滑 | 车辆框、区域车辆数、车辆密度、拥堵等级、持续时间、热力图数据 | 待真实数据验证 |
| 禁停区域车辆跟踪 | YOLO 车辆检测 + ByteTrack 目标跟踪 + 禁停区域时空规则 | 跟踪编号、车辆位置、车辆轨迹、进入时间、停留时长、禁停事件和告警 | 待真实数据验证 |
| 道路障碍与遗撒物检测 | FastFlow 异常区域检测 + 道路区域约束 + 异常面积与持续时间规则 | 异常区域、异常分数、出现时间、持续时间、受影响车道和告警等级 | 待真实数据验证 |
| 智能分析报告 | 外部多模态大模型 API + 结构化任务上下文 + 关键事件截图 + 固定报告模板 | 任务摘要、交通状态分析、重点事件分析、趋势解释、综合结论和处置建议 | 待接口与报告质量验证 |

候选路线验证失败时调整对应模块设计，不降低业务需求。

## 10. 用户与权限

系统设置管理员和监控员两类固定角色。

管理员负责用户、设备、场景模板、模型、区域、参数范围、白名单、视频延期、异常任务强制终止和报告审核归档。监控员负责实时任务创建与结束、已有模板选择、模板允许的视觉功能组合与任务参数调整、实时查看、交通告警处理、历史查询、历史视频查看、关键截图复核、报告草稿生成、提交审核和生成失败重试。管理员通过与监控员相同的任务接口保留任务创建与结束权限，用于管理和异常干预。

移动 Web 边端通过设备注册码完成首次注册，注册码由管理员创建和管理。边端设备注册不使用管理员或监控员的交互账号。

管理员和监控员均可查看和导出当前有效归档报告。关键操作记录执行用户和执行时间。

## 11. 交通告警

交通告警包括未授权车辆、持续拥堵、禁停超时和道路障碍四类。设备离线、视频中断和算法故障作为系统运行异常管理，不进入交通告警统计。

告警等级包括低、中、高三级，由各业务可测量指标和场景模板阈值通过确定性规则计算。大模型不参与告警触发和等级判断。

告警处理状态包括待处理、处理中和已处理。待处理告警由首个成功领取的用户进入处理中，处理中告警由当前处理人按照预期数据版本提交处理结果。管理员可以改派处理人，改派操作保留原处理人、新处理人、原因和时间。处理记录包含处理人、开始处理时间、处理结果和完成时间。

同一目标或同一区域的连续同类告警应合并为可持续更新的业务事件，不按分析帧重复创建告警。

## 12. 实时结果同步

前端采用“最新结果直接显示 + 过期结果过滤”的方式叠加结构化分析结果。

每条结果至少包含任务编号、视频源编号、帧序号、视频帧时间、处理完成时间、源视频尺寸和业务结果。前端过滤非当前任务、乱序和超过有效时间的结果。

结果有效时间、视频播放性能、算法分析帧率、结果刷新率、处理延迟和告警延迟在真实环境实践后确定并冻结。

## 13. 智能分析报告

智能分析报告模块只面向已结束且数据完整的单次实时任务。系统不建设通用聊天、运行诊断、自然语言查询、工具调用和系统操作功能。

报告事实部分由云端模板生成，包括任务概况、配置摘要、业务统计、告警统计、关键事件、截图和性能汇总。外部多模态大模型只生成任务摘要、交通状态分析、重点事件分析、趋势解释、综合结论和处置建议。

外部调用遵循最小必要原则。设备使用内部编号，车牌默认脱敏，只提交任务统计、事件摘要、性能摘要和监控员复核后的关键截图，不提交完整视频、用户凭证、控制指令和无关历史数据。

报告状态包括生成中、草稿、待审核、已归档、已驳回、已撤销和生成失败。监控员生成草稿并提交审核，管理员归档或驳回。已驳回版本保留驳回意见并停止进入归档流程；监控员重新生成新的草稿版本。生成失败记录可以由监控员重新发起生成，新请求形成新版本。

同一任务同时只能存在一个有效归档报告。更新正式报告时，系统先校验新版本满足归档条件，再在同一数据库事务中将旧版本更新为已撤销并将新版本更新为已归档，保证替换操作完成后仍只有一个有效归档版本。

只有已归档报告可以导出 PDF。已撤销版本保留内容、PDF 和撤销记录，用于审计。

## 14. 数据与文件生命周期

MySQL 保存用户、设备、场景模板、模型、任务、配置快照、逐帧分析结果、长期业务记录、交通告警、性能汇总、操作记录、报告和文件元数据。

云端本地磁盘保存完整任务视频、关键截图和报告 PDF。前端通过云端受控接口访问文件，不直接访问磁盘路径。

完整任务视频和普通逐帧分析结果从任务结束起默认保存 7 天；异常任务从资源清理完成或管理员强制终止时开始计算。管理员延长完整任务视频保留期限时，同步延长该任务普通逐帧分析结果的保留期限，保证视频与逐帧结果可以共同复核。关键截图、业务事件、交通告警、统计、操作记录和报告长期保存。

普通逐帧分析结果是未被提升为车牌记录、拥堵时间序列、禁停事件、道路异常事件、交通告警或关键截图依据的帧级目标、区域状态和算法分数。到期清理前，系统已经将需要长期使用的数据独立保存为对应业务记录。

长期保存表示系统不通过常规到期任务自动删除这些数据。正式档案只有管理员通过经审计的数据删除操作才能删除；具体备份周期、容量告警和恢复流程在系统设计报告中确定。

视频到期删除后保留文件元数据和删除时间，不级联删除任务、事件、截图和报告。

## 15. 数据采集与验收指标

真实数据按照开发、训练或适配、验证和测试用途分开管理，并以独立采集批次划分。测试数据不得参与模型训练、参数调整和阈值选择。

四类专题场景均采集正常样本、目标事件样本和干扰样本。人工真值根据业务分别包含车牌字符、区域车辆数、车辆轨迹与停留时间、道路异常区域和告警触发时间。

当前阶段只固定指标名称、数据条件、计算方式和阈值确定流程。准确率、帧率、连接时限、处理延迟、结果有效时间、告警延迟、故障判定时间、资源阈值和允许的综合功能组合均在实践后冻结，并同步写入需求、设计和测试文档。冻结记录应包含测试环境、数据集版本、测量方法、结果、最终阈值和确认日期。

阈值冻结后不得根据最终测试结果临时降低。

## 16. 前端页面

系统主要页面包括登录、实时监控、任务管理、设备管理、场景与分析配置、白名单管理、告警管理、历史查询、报告管理和系统管理。

系统采用一个统一实时监控页面。闸机、拥堵、禁停和道路异常作为专题分析面板，根据当前任务实际启用功能显示；综合监控可同时显示多个可切换面板。

## 17. 开发优先级

开发优先级用于安排实施顺序，不改变最终验收范围：

1. 高优先级：登录与权限、移动 Web 边端、任务控制、WebRTC 视频上行与播放下行、实时监控、任务状态、MySQL 基础数据和任务视频留存。
2. 中优先级：四类视觉业务、五类场景模板、交通告警、历史查询、系统监控和数据生命周期管理。
3. 低优先级：智能分析报告、报告审核版本、PDF 导出、页面体验和运维完善。

以上范围内的所有需求均需在 V2.0 最终验收前完成。

## 18. 一致性约束

需求分析报告负责说明业务目标、用户、功能、数据、非功能要求和验收标准，不固定未经验证的模型版本、软件库版本和实现参数。

系统设计报告负责说明架构、模块、候选技术、接口、数据结构、部署、错误处理和测试方法。完成实践后，将最终算法选型和冻结阈值补入系统设计与需求基线。

需求、设计、实现和测试必须保持以下对应关系：每项功能需求具有设计模块和验收用例；每项长期数据具有来源、存储和查询入口；每种任务或报告状态具有进入条件和退出条件；每种交通告警具有触发依据、等级规则、处理流程和历史记录。
