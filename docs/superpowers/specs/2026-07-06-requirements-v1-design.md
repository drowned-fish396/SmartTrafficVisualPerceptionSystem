# 需求分析报告 V1.0 写作设计说明

## 1. 文档目标

本文档用于指导《需求分析报告V1.0》的正式编写。报告面向软件工程实训立项评审，重点说明“云边端协同的智慧交通视觉感知系统”要解决的问题、用户角色、业务场景、功能范围、非功能需求、关键技术取舍、需求优先级和验收标准。

V1.0 阶段不追求完整实现细节，而是优先把项目边界和可演示闭环讲清楚，为后续系统设计方案、V2.0 需求分析、代码实现和验收演示提供依据。

## 2. 已确认的核心决策

| 决策项 | V1.0 写法 |
|---|---|
| 报告版本 | 需求分析报告V1.0 |
| 默认演示场景 | 学院704智慧交通沙盘 |
| 视频输入形态 | 实时视频流为主，视频段上传为兜底 |
| 数据传输组合 | RTMP/RTSP 视频流 + REST API + WebSocket |
| 云边端通讯 | 边端到云端传视频流和心跳，云端到前端推送分析结果，前端到云端提交查询和配置 |
| 实时视频流 AI 分析 | 云端持续拉流或接收推流，按帧抽取并进行 AI 推理，实时推送结构化结果 |
| 处理效率目标 | 课程演示型实时，端到端延迟 1-3 秒，云端分析帧率 5-15 FPS |
| AI 分析准确度 | 面向 704 沙盘清晰画面达到课程演示可用，不承诺生产级全场景准确率 |
| 模型策略 | 轻量检测模型 + OCR/规则 + 模拟兜底 |
| 写作策略 | 演示闭环优先型 |

## 3. 报告章节结构

### 3.1 引言

说明课程背景、项目来源、704智慧交通沙盘场景、云边端协同目标和报告编写目的。

### 3.2 项目概述

说明系统由手机边端采集设备、云端分析服务、前端展示端组成。核心目标是完成交通视频采集、传输、AI 分析、实时展示、管理配置和历史查询。

### 3.3 用户角色与使用场景

需要覆盖边端设备使用者、平台管理员、算法/模型配置人员、评审/演示观看者，并给出 704 沙盘下的典型演示流程。

### 3.4 业务需求

围绕四类核心视觉感知任务组织：车牌识别与白名单比对、车辆识别与拥堵热力图、禁停区域车辆跟踪、道路异常检测。

### 3.5 功能需求

按模块拆分：边端采集、云端视频接入、云边端通讯、实时视频流 AI 分析、设备管理、模型管理、系统管理、监控展示、历史查询与统计。

### 3.6 界面与展示需求

明确前端页面和展示内容，使需求能直接映射到后续前端实现和演示验收。建议页面包括实时监控大屏、闸机监控视图、拥堵热力图视图、禁停区域监控视图、道路异常监控视图、设备管理页面、模型管理页面、系统状态页面、历史查询页面。

### 3.7 关键问题需求决策

单独回答六个核心问题：视频流还是视频段、数据如何传输、云边端如何通讯协同、实时视频流如何进行 AI 分析、处理时间效率如何设定、AI 分析准确度和模型策略如何设定。每个问题都应写成“主方案 + 兜底方案 + 选择理由”。

### 3.8 非功能需求

覆盖实时性、通讯稳定性、AI 分析准确度、可演示性、可扩展性、可维护性、数据持久化、安全性和兼容性。

### 3.9 数据需求

明确设备数据、视频源数据、检测结果、告警事件、模型配置、白名单车辆库、历史统计数据的结构化要求。

### 3.10 需求优先级建议

按照 P0 必须完成、P1 优先完成、P2 可选增强三档划分需求，说明每档服务于哪些课程目标和演示目标。

### 3.11 系统边界与约束

说明不要求真实控制闸机、不要求生产级低延迟、不强制训练自有大模型、不提交大视频和大模型权重，重点保证可演示闭环。

### 3.12 验收标准

按课程评分标准写可检查结果，例如手机能推流或上传视频、边端设备能注册和上报心跳、云端能持续接收视频并进行 AI 分析、前端能实时展示检测框/车牌/热力图/告警、历史数据可查询。验收标准应包含通讯链路验收、实时分析验收和准确度演示验收。

### 3.13 风险与应对

覆盖网络不稳定、云边端通讯中断、模型效果不足、AI 分析准确度不稳定、硬件算力不足、704 场景数据不足、实时性达不到目标等风险，并给出视频段兜底、心跳重连、模拟结果兜底、轻量模型和区域规则等应对措施。

### 3.14 后续迭代方向

说明 V2.0 将进一步细化接口、数据结构、系统设计方案、测试指标和演示流程。

## 4. 需求优先级原则

P0 必须保证系统从边端输入到云端分析再到前端展示形成最小闭环。P1 覆盖课程要求中的四类核心视觉感知任务和关键管理能力。P2 用于增强展示效果、系统完整性和创新性，不影响 V1.0 立项评审的基本成立。

## 5. 三个重点增强方向

### 5.1 云边端通讯

正式报告应把云边端通讯写成可验证的数据链路：边端手机向云端传输实时视频流并上报设备注册和心跳；云端向前端推送 AI 分析结果、告警事件和系统状态；前端向云端提交历史查询、模型配置和区域配置请求。V1.0 阶段不强制实现云端对边端的复杂反向控制，但可以列为扩展能力。

### 5.2 实时视频流 AI 分析

正式报告应明确系统支持对实时视频流进行持续 AI 分析。云端从 RTMP/RTSP 视频流中抽帧，调用车辆检测、车牌识别、目标跟踪、拥堵分析和异常检测模块，生成结构化分析结果，并通过 WebSocket 推送至前端。为保证课程演示稳定性，实时流失败时允许视频段上传作为兜底。

### 5.3 AI 分析准确度

正式报告应采用“课程演示准确度”而非“生产级准确率”表述。准确度要求限定在 704 沙盘、光照稳定、摄像角度固定、目标无遮挡或轻微遮挡的条件下。车辆检测应能稳定发现主要车辆目标，车牌识别应能对清晰车牌输出结果，拥堵热力图应能反映车辆密度趋势，禁停跟踪应能计算区域内停留时长，道路异常检测应能识别预设异常物体或使用模拟异常事件兜底。

## 6. 写作约束

正式报告应避免只写单机算法或纯前端展示，应突出手机、云端、前端三端协同。需求描述应可验证、可演示、可追踪，并为后续系统设计和开发拆分提供依据。
