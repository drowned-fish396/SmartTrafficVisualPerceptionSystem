# 系统设计业务问题说明

## 1. 文档目的

本文档用于单独解释《系统设计方案 V1.0》中部分设计决策背后的业务原因和工程原因。

这些内容不直接写入系统设计方案正文，避免正式设计文档过于解释性和口语化。本文档可作为团队内部理解材料，也可用于答辩准备、设计讨论和后续修改依据。

## 2. 为什么系统不能只做单机视频分析

如果系统只做单机视频分析，最简单的形式是使用一个 Python 脚本读取本地视频，然后输出车辆检测、车牌识别或告警结果。这种方式适合验证算法，但不适合作为本项目的完整系统方案。

本项目主题是“云边端协同的智慧交通视觉感知系统”，核心不只是识别视频中的车辆，而是模拟真实交通感知平台中的完整链路。真实系统通常需要边端设备采集视频，云端服务接收和分析视频，前端平台展示结果和管理状态。如果只做单机脚本，系统会缺少设备接入、视频传输、任务状态、前端展示、历史查询和系统管理等关键环节。

因此，单机视频分析可以作为算法模块的一部分，但不能替代完整系统。系统设计需要将算法能力嵌入云端服务，并与边端采集、前端展示和数据存储形成闭环。

## 3. 为什么要设计设备注册和心跳

设备注册用于让云端知道有哪些边端设备接入系统，设备心跳用于让云端持续判断设备是否在线。

在真实平台中，视频源通常来自多个摄像头、手机或边缘设备。云端不能只接收一段视频，而需要知道视频来自哪个设备、设备是否在线、最近是否上报过状态、当前视频源是否正常。如果没有设备注册和心跳，系统就无法支撑设备管理页面，也无法解释视频中断、设备离线或任务异常的原因。

在课程项目中，即使只使用一台手机演示，也建议保留设备注册和心跳设计。这样可以体现边端设备与云端服务之间的协同关系，也能让设备管理页面具有真实含义。

## 4. 为什么需要视频上传兜底

实时推流更能体现系统实时性，但它依赖手机、局域网、推流工具、视频服务和云端拉流能力。课程演示环境中，网络不稳定、手机权限限制、推流软件配置错误或端口占用都可能导致实时流失败。

视频上传兜底的作用是保证在实时推流不可用时，系统仍然可以接收交通场景视频，并进入同一套云端分析和前端展示流程。这样即使实时性降低，系统仍能完成“边端输入、云端分析、前端展示、历史查询”的基本闭环。

因此，V1.0 可以将实时推流作为主方案，将短视频上传作为兜底方案。这样既能体现实时系统设计，也能降低验收演示风险。

## 5. 为什么模型不可用时要有模拟推理

AI 模型运行受模型文件、依赖环境、硬件性能、视频质量和场景差异影响。课程项目中，模型可能出现加载失败、识别效果差、推理速度慢或某些场景无法稳定识别的问题。

模拟推理并不是为了替代真实模型，而是为了保证系统演示链路稳定。当模型不可用时，系统可以基于预设样例、固定时间点、配置区域或测试数据生成结构化检测结果和告警事件。前端仍然可以展示检测框、车牌结果、热力图和告警列表，数据库仍然可以记录历史数据。

这种设计能够将“系统工程能力”和“算法效果不稳定”解耦。真实模型用于展示项目技术能力，模拟推理用于保证系统闭环和演示可靠性。

## 6. 为什么要把检测结果结构化存储

检测结果如果只显示在画面上，系统就无法支持历史查询、统计分析、告警追溯和测试验证。结构化存储可以将每次检测和告警转化为数据库中的记录，便于后续查询和分析。

结构化检测结果通常需要包含时间戳、设备 ID、任务 ID、目标类别、目标位置、置信度、事件类型、区域 ID 和可视化字段。这些字段既能支持前端实时展示，也能支持历史查询和统计图表。

对于本项目来说，结构化存储还能增强文档和系统的一致性。需求分析中提到车牌数量、拥堵趋势、违规停车数量和异常事件查询，系统设计中就必须有对应的数据结构和数据库表来支撑。

## 7. 为什么要区分实时数据和历史数据

实时数据用于当前监控展示，强调低延迟和持续刷新；历史数据用于查询、统计和追溯，强调持久化、可筛选和可复现。

例如，车辆检测框和当前拥堵等级属于实时展示数据，可以通过 WebSocket 推送给前端；车牌识别记录、禁停告警、异常事件和拥堵趋势则需要写入数据库，供历史查询页面使用。如果不区分二者，系统容易出现前端只能看当前画面、但无法查询过去记录的问题。

因此，系统应采用“实时推送 + 关键数据落库”的方式。实时通道服务于监控展示，数据库服务于历史查询和统计分析。

## 8. 为什么前端要有设备管理、模型管理和系统状态

如果前端只展示视频和检测框，系统会更像一个算法演示页面，而不是智慧交通视觉感知平台。

设备管理用于展示边端设备是否在线、视频源是否正常和最近心跳时间，体现边端接入能力。模型管理用于展示当前启用的算法模型、阈值配置、检测区域和禁停区域，体现 AI 分析任务可配置能力。系统状态用于展示 CPU、GPU、内存、视频传输和分析任务状态，体现平台运行监控能力。

这些管理页面不一定在 V1.0 做得很复杂，但需要作为系统设计的一部分保留。它们能够证明系统不是单一算法脚本，而是具备平台管理能力的云边端协同系统。

## 9. 为什么 V1.0 不适合上完整微服务架构

完整微服务架构通常需要服务拆分、服务注册、网关、配置中心、日志链路、容器编排和服务治理。对于课程项目来说，这会显著增加开发、部署和联调成本，而且很多复杂能力在 V1.0 演示中并不会真正发挥作用。

V1.0 更适合采用模块化单体后端。也就是说，设备管理、视频接入、分析任务、模型配置、历史查询和 WebSocket 推送可以在同一个后端项目中实现，但在代码结构上保持清晰模块边界。这样既降低了实现难度，又为后续拆分微服务保留了空间。

如果后续系统规模扩大，可以优先将 AI 推理、视频处理和实时推送拆分为独立服务。

## 10. 为什么 SQLite 对课程演示足够，但要保留 PostgreSQL 扩展方向

SQLite 部署简单，不需要单独安装数据库服务，适合课程项目的本地开发、演示和测试。对于少量设备、少量视频源和有限历史数据的 V1.0 场景，SQLite 能够满足基本存储需求。

但是，SQLite 不适合高并发、多用户、大规模数据写入和复杂权限控制。如果系统后续扩展到多设备持续接入、多任务并行分析和大量历史数据查询，就更适合使用 PostgreSQL 这类服务端数据库。

因此，V1.0 可以采用 SQLite 作为默认数据库，同时在设计中保持数据访问层独立，避免业务逻辑直接绑定具体数据库。这样后续从 SQLite 切换到 PostgreSQL 时，改动范围会更可控。
