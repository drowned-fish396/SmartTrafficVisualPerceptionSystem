# 云边端协同的智慧交通视觉感知系统需求分析报告 V2.0

## 1. 引言

### 1.1 项目背景

本项目面向学院 704 智慧交通沙盘，建设由移动 Web 边端、云端服务和前端展示端组成的智慧交通视觉感知系统。系统通过手机采集真实交通视频，由云端完成视频接入、视觉分析、业务事件处理和数据存储，并由前端完成实时监控、平台管理、历史查询和分析报告展示。

项目以云边端协同为核心。边端负责现场视频采集，云端负责视频处理和智能分析，前端负责业务操作和结果展示，三端通过明确的数据协议共同完成实时任务。

V2.0 在 V1.0 的视觉感知和平台管理范围上重新梳理任务流程、视频链路、用户权限、交通告警、数据生命周期和验收口径，并增加基于外部多模态大模型的智能分析报告功能。

### 1.2 编写目的

本文档用于明确系统 V2.0 的建设目标、业务范围、用户角色、任务流程、功能需求、数据需求、界面需求、非功能需求、开发优先级和验收标准，作为系统设计、开发实现、系统测试和课程验收的依据。

### 1.3 文档范围

本文档覆盖以下内容：

1. 移动 Web 边端的设备注册、任务指令接收、摄像头采集和 WebRTC 视频传输。
2. 云端的任务管理、视频接入与播放服务、任务视频留存、视觉分析、业务规则、交通告警、数据存储、历史查询和智能分析报告。
3. 前端的登录认证、实时监控、设备管理、场景与分析配置、告警处理、历史查询和报告管理。
4. 车牌识别与白名单比对、车辆识别与拥堵分析、禁停区域车辆跟踪、道路障碍与遗撒物检测四类视觉业务。
5. 用户权限、数据真实性、数据生命周期、系统运行环境和验收要求。

### 1.4 术语说明

| 术语 | 说明 |
| --- | --- |
| 移动 Web 边端 | 运行在手机浏览器中的采集页面，负责设备注册、任务指令接收、摄像头采集和视频上行 |
| 云端 | 部署在云端电脑上的业务、视频、算法、数据和报告服务 |
| 前端展示端 | 部署在独立电脑上的监控与管理 Web 应用 |
| 实时分析任务 | 由前端创建、云端管理、边端执行视频采集并由云端完成实时分析的业务任务 |
| 场景模板 | 对监控场景、视觉功能组合、区域、模型参数、业务阈值和展示内容的预设配置 |
| 结构化分析结果 | 云端从视频帧中生成的目标位置、置信度、业务状态、事件和可视化字段 |
| 任务视频留存 | 云端在实时任务期间保存所接收视频并形成可回放文件的功能 |
| 完整任务视频 | 任务视频留存功能生成并与任务关联的视频文件 |
| 逐帧分析结果 | 云端对单个采样视频帧产生的目标、区域和算法输出，用于实时展示、事件计算和短期复核 |
| 长期业务记录 | 由逐帧分析结果聚合或确认形成的车牌记录、拥堵时间序列、禁停事件、道路异常事件、交通告警、统计和关键截图 |
| 交通告警 | 由确定性业务规则生成的未授权车辆、持续拥堵、禁停超时或道路障碍事件 |
| 智能分析报告 | 基于单次已结束任务的事实数据、关键截图和大模型分析性正文形成的任务报告 |

## 2. 项目概述

### 2.1 建设目标

系统应形成以下完整业务链路：

1. 手机通过移动 Web 边端注册设备并保持在线。
2. 监控员通过前端创建实时分析任务。
3. 云端向指定边端下发任务启动指令。
4. 边端采集视频并通过 WebRTC 发送至云端。
5. 云端通过 WebRTC 向前端提供实时视频，同时获取视频帧执行视觉分析。
6. 云端通过 WebSocket 推送结构化分析结果，前端在实时视频上叠加标注。
7. 云端执行任务视频留存，并保存业务结果、交通告警和性能数据。
8. 用户查询历史任务、处理交通告警并生成智能分析报告。

### 2.2 系统组成

| 组成部分 | 角色定位 | 主要职责 |
| --- | --- | --- |
| 移动 Web 边端 | 现场采集执行端 | 设备注册、摄像头授权、任务指令接收、视频采集、WebRTC 视频上行和状态上报 |
| 云端服务 | 任务、视频、算法和数据中心 | 用户认证、设备控制、任务管理、WebRTC 视频服务、任务视频留存、视觉分析、业务规则、交通告警、数据存储、历史查询和智能分析报告生成 |
| 前端展示端 | 监控与管理入口 | 登录、任务控制、实时视频播放、结果叠加、平台管理、告警处理、历史查询和报告管理 |

智能分析报告模块部署在云端服务内部。该模块根据任务事实数据和关键事件截图组织报告内容，并通过 HTTPS API 调用外部多模态大模型生成分析性正文。外部大模型属于云端依赖的外部技术服务，不作为与边端、云端和前端并列的系统组成部分。

### 2.3 建设范围

V2.0 只建设实时分析任务。同一时刻只允许一个任务处于已创建、启动中、运行中或结束中；尚未完成活动资源清理的异常任务同样占用全局任务名额。

系统支持闸机通行监控、道路交通态势、禁停区域监控、道路异常监控和综合监控五类场景模板。监控员创建任务时选择已有场景模板，在模板允许范围内调整视觉功能和本次任务参数。

### 2.4 运行场景约束

V2.0 以学院 704 智慧交通沙盘作为正式开发、测试和验收场景。手机可以在不同任务之间调整拍摄位置，但单次任务开始前必须使用支架或其他方式固定手机，并完成画面、场景模板和监控区域确认。任务运行期间摄像头位置和方向应保持稳定。

闸机、道路、禁停区域和道路异常可以使用不同机位。综合监控只启用当前机位具备有效观测条件的视觉功能。

系统应记录关键操作的操作人、操作对象、操作类型、操作时间和操作结果。

移动 Web 边端使用管理员创建的设备注册码完成首次注册，不使用管理员或监控员的交互账号。

## 4. 核心业务流程

### 4.1 边端准备流程

1. 手机通过 HTTPS 打开移动 Web 边端页面。
2. 用户输入有效设备注册码完成设备注册。
3. 用户授予浏览器摄像头权限。
4. 边端与云端建立控制连接并上报设备状态。
5. 设备满足页面在线、摄像头权限有效且未执行其他任务时进入空闲状态。

### 4.2 任务启动流程

1. 监控员或管理员在前端选择在线空闲设备。日常任务操作由监控员承担。
2. 操作用户选择已有场景模板和模板允许的视觉功能组合，并在管理员设置的范围内调整本次任务参数。
3. 前端向云端提交任务创建并启动请求。
4. 云端在同一数据库一致性操作中校验全局任务名额、设备状态和任务配置。
5. 校验通过后，云端创建任务、保存配置快照并原子占用全局任务名额。
6. 云端立即向边端发送启动采集指令。
7. 边端启动摄像头并建立 WebRTC 视频上行会话。
8. 云端接收视频，启动任务视频留存，准备前端 WebRTC 播放会话并启动所选视觉功能。
9. 云端确认视频和分析流程正常后将任务更新为运行中。
10. 前端播放实时视频并叠加最新有效分析结果。

### 4.3 任务结束流程

1. 监控员或管理员通过前端提交结束任务请求。日常任务结束操作由监控员承担。
2. 云端将任务更新为结束中，并向边端发送结束指令。
3. 边端停止摄像头采集和 WebRTC 视频上行。
4. 云端停止获取新的分析输入帧，完成已进入处理流程的数据保存。
5. 云端关闭视频上行与播放下行会话，停止任务视频留存并完成视频文件封装。
6. 云端汇总业务统计、交通告警和系统性能数据。
7. 云端执行任务数据完整性检查并记录检查结果。
8. 云端完成任务收尾后将任务更新为已结束，并释放设备与全局任务名额；未通过完整性检查的任务不能生成报告，管理员可以重新执行数据汇总。

### 4.4 告警处理流程

1. 云端根据业务规则生成交通告警。
2. 前端按照告警等级和发生时间展示告警。
3. 管理员或监控员领取待处理告警，首个领取成功的用户成为当前处理人，告警更新为处理中。
4. 当前处理人完成处理后，按照预期数据版本提交处理结果并将告警更新为已处理；管理员需要更换处理人时，应先完成带版本校验的改派操作。
5. 告警及其证据、等级和处理记录进入历史查询和报告数据范围。

### 4.5 报告生成流程

1. 监控员选择已结束且通过数据完整性检查的任务。
2. 云端汇总任务事实数据并自动选择候选关键截图。
3. 监控员复核候选截图。
4. 云端构建最小必要上下文并调用外部大模型。
5. 云端将固定事实部分与大模型分析性正文组合为报告草稿。
6. 监控员检查草稿并提交管理员审核。
7. 管理员归档合格报告，或填写原因驳回当前版本。
8. 已归档报告可以查看和导出 PDF。

## 5. 状态需求

### 5.1 任务状态

| 状态 | 定义 |
| --- | --- |
| 已创建 | 云端已创建任务、保存配置快照并占用全局任务名额，准备立即启动边端和视频流程 |
| 启动中 | 云端正在下发边端指令，并建立视频上行、播放下行、任务视频留存和算法流程 |
| 运行中 | 边端正在采集，云端持续接收和提供视频，并执行所选视觉功能 |
| 结束中 | 云端正在停止边端、关闭视频与算法流程并汇总任务数据 |
| 已结束 | 任务采集和分析流程已经正常停止，视频封装与任务收尾已经完成 |
| 异常 | 阻断性故障导致任务无法继续完成实时监控或视觉分析目标 |

正常状态迁移为：已创建、启动中、运行中、结束中、已结束。启动中、运行中和结束中发生阻断性故障时可以进入异常。

任务创建成功后应自动进入启动流程，不提供长期保留在已创建状态的人工操作。创建前校验失败时不生成任务记录；创建后启动失败时任务进入异常。

异常任务应通过当前处理阶段记录自动清理、清理重试和清理失败。完成边端采集停止、视频上行与播放下行会话关闭、算法进程停止和任务视频写入停止后，系统释放设备与全局任务名额。自动清理失败时，管理员可以执行强制终止；强制终止必须确认上述四类活动资源均已停止，随后释放设备与全局任务名额。任务视频文件封装修复、数据汇总和文件一致性检查等未完成数据处理项可以转入后续修复流程，并记录异常与操作日志。

### 5.2 边端状态

| 状态 | 定义 |
| --- | --- |
| 离线 | 云端无法与边端建立有效控制连接 |
| 空闲 | 边端在线、摄像头权限有效且当前未执行任务 |
| 任务执行中 | 边端正在为指定任务采集和发送视频 |
| 异常 | 权限、摄像头或网络问题导致边端无法执行当前任务 |

### 5.3 告警处理状态

交通告警处理状态包括待处理、处理中和已处理。

### 5.4 报告状态

| 状态 | 定义 |
| --- | --- |
| 生成中 | 云端正在组织任务数据并调用外部大模型 |
| 草稿 | 报告已生成，尚未提交管理员审核 |
| 待审核 | 报告已提交管理员审核 |
| 已归档 | 报告审核通过并成为当前正式版本 |
| 已驳回 | 管理员已驳回当前版本，该版本保留审核意见且不能直接归档 |
| 已撤销 | 原正式报告已被管理员撤销并保留审计记录 |
| 生成失败 | 数据组织、大模型调用或报告结构校验失败 |

管理员驳回待审核报告时，应将该版本更新为已驳回并记录驳回意见。监控员根据驳回意见生成新的草稿版本。

同一任务同时只能存在一个有效归档报告。更新正式报告时，系统应在一次一致性操作中撤销旧版本并激活新版本。

### 5.5 任务数据完整性

任务数据完整性检查包括公共必检项和按启用功能检查项。

公共必检项包括任务状态日志、任务配置快照、视频文件封装与校验状态、算法运行记录、任务统计汇总和性能汇总。

按启用功能检查项包括相应业务处理记录、业务统计和告警合并结果。某项已启用功能在任务期间没有检测到目标或事件时，应记录合法的零结果；没有运行记录或汇总记录属于数据缺失。

数据完整性检查结果应包含通过状态、检查时间、检查项和缺失项。未通过检查的任务可以查看已有数据和视频，但不能生成报告。

### 7.2 统一实时监控页面

实时监控页面采用统一主视图，不建设四个相互独立的一级监控页面。

页面应保持视频、任务状态和当前告警可见，并根据任务实际启用功能显示以下专题面板：

1. 闸机通行面板：车牌号码、置信度、图像质量、白名单状态和通行判断。
2. 道路交通态势面板：区域车辆数、车辆密度、拥堵等级、持续时间和热力图。
3. 禁停区域面板：跟踪编号、车辆轨迹、进入时间、停留时长和告警状态。
4. 道路异常面板：异常区域、异常分数、持续时间、受影响车道和告警等级。

综合监控任务应按照实际启用功能显示多个可切换专题面板。

### 7.3 结果叠加要求

前端应在 WebRTC 视频播放层上方设置独立标注层。标注层应根据源视频尺寸、页面缩放比例和留白偏移换算目标坐标。

前端采用最新结果直接显示和过期结果过滤策略。视频下行和分析结果使用云端输入帧的统一视频会话编号、帧序号和媒体时间作为共同时间基准。旧任务、旧视频会话、乱序和超过有效时间的结果不得覆盖当前画面；结果过期、任务切换或播放重连时应清除旧标注。

## 8. 数据需求

### 8.1 数据真实性与关联

每项任务、分析结果、业务事件、交通告警、关键截图、性能汇总和报告均应关联相应的任务编号、设备编号和时间信息。

系统应区分逐帧分析结果、长期业务记录、确定性业务规则结果和大模型分析性正文，不得将大模型生成内容写回原始检测或告警事实字段。

### 8.2 用户和设备数据

用户数据应包含用户编号、账号、密码摘要、角色、账号状态、创建时间和更新时间。

设备数据应包含设备编号、设备名称、设备注册码关联、浏览器与系统信息、注册时间、在线状态、最近心跳、摄像头权限状态、当前任务和说明。

### 8.3 场景和任务数据

场景模板应包含模板编号、模板名称、版本、适用场景、默认功能、可选功能、模型配置、区域配置、业务阈值、告警分级规则和展示配置。

任务数据应包含任务编号、任务名称、设备、场景模板、配置快照、创建人、创建时间、开始时间、结束时间、任务状态、当前阶段、异常信息和完整性检查结果。

### 8.4 视觉与业务数据

逐帧分析结果应包含任务、视频会话、帧序号、媒体时间、处理时间、源视频尺寸、目标类别、目标位置、置信度和模型版本。逐帧分析结果包括目标检测、区域状态和算法分数，不包括已经独立保存的长期业务记录。

长期业务数据应包含车牌记录、拥堵时间序列、禁停事件、道路异常事件、交通告警、关键截图和任务级统计。

### 8.5 告警数据

交通告警应包含告警编号、类型、等级、处理状态、当前处理人、数据版本、关联任务、设备、区域、首次触发时间、最近更新时间、持续时间、业务依据、关键截图和处理记录。

### 8.6 报告数据

报告应包含报告编号、任务、版本号、状态、事实数据快照、输入摘要、截图清单、分析性正文、大模型服务标识、生成用户、生成时间、审核记录、撤销记录和 PDF 文件信息。

### 8.7 文件数据

用户、设备、任务、配置、逐帧分析结果、长期业务记录、告警、审计记录、报告和文件元数据保存于 MySQL。完整任务视频、关键截图和报告 PDF 保存于云端本地磁盘。系统应保存文件编号、类型、关联对象、路径、格式、大小、校验值、创建时间、到期时间和文件状态。

前端通过云端权限校验接口访问文件，不直接访问本地磁盘路径。

### 8.8 数据生命周期

完整任务视频和普通逐帧分析结果从任务进入已结束状态开始默认保存 7 天。异常任务从资源清理完成或管理员强制终止的时间开始计算 7 天保留期。

管理员延长完整任务视频保留期限时，应同步延长该任务普通逐帧分析结果的保留期限，并记录操作人、时间和原因。

关键截图、车牌记录、拥堵时间序列、禁停事件、道路异常事件、交通告警、任务统计、性能汇总、操作记录和报告不通过常规到期任务自动删除。

视频到期删除后，应保留文件元数据、到期时间和实际删除时间，不得级联删除任务、事件、截图或报告。

## 9. 外部接口与通信需求

| 通信方向 | 主要内容 | 方式 |
| --- | --- | --- |
| 手机边端与云端 | 页面访问、注册、控制连接和状态上报 | HTTPS、WSS |
| 手机边端到云端 | 实时视频上行 | WebRTC |
| 云端到前端 | 实时视频播放 | WebRTC |
| 云端到前端 | 任务状态、结构化结果和交通告警 | WebSocket |
| 前端与云端 | 登录、配置、任务、查询、告警和报告操作 | REST API |
| 云端与外部大模型 | 最小必要任务上下文和报告分析结果 | HTTPS API |

所有外部接口应使用明确的数据结构、错误码和权限校验。视频会话、结构化结果和业务操作应通过任务编号保持关联。

## 10. 非功能需求

### 10.1 真实性

系统应使用真实采集视频执行视觉分析，检测结果、统计、交通告警和报告事实均应能够追溯到对应任务和视频数据。

### 10.2 安全性

移动 Web 边端和前端应通过受信 HTTPS 域名访问系统。用户密码应以安全摘要保存，登录凭证应具有有效期，管理接口应执行角色校验。

外部大模型调用应遵循最小必要数据原则。设备使用内部编号，车牌默认脱敏，完整视频、登录凭证和控制指令不得发送至外部服务。

### 10.3 可靠性

系统应识别并记录边端、WebRTC、算法、数据库、文件和大模型调用异常。短时波动不直接终止任务，持续性阻断故障应使任务进入异常并执行资源清理。

### 10.4 可追溯性

任务状态变化、配置快照、视觉结果、业务事件、交通告警、关键截图、视频、报告版本和关键操作应形成完整关联。

### 10.5 可维护性

边端、任务、视频、算法、告警、数据和报告模块应具有清晰职责与稳定接口。视觉算法可以在不改变业务输出契约的前提下替换。

### 10.6 可扩展性

系统应支持后续增加场景模板、视觉功能和设备数量。V2.0 的单任务限制和本地部署不应导致业务数据结构与单一设备硬编码绑定。

### 10.7 兼容性

移动 Web 边端应在选定的手机浏览器中完成摄像头授权和 WebRTC 视频上行。前端展示端应在选定桌面浏览器中完成 WebRTC 播放、WebSocket 接收和标注绘制。

### 10.8 运行环境

云端运行环境为 Windows 11、AMD Ryzen 9 7945HX、32 GB 内存、NVIDIA GeForce RTX 4060 8 GB 显存和约 400 GB 可用 SSD 空间。云端业务、视频、算法、数据和文件服务部署在同一台电脑，前端部署在另一台电脑。

### 10.9 性能指标冻结要求

当前阶段固定性能指标名称和测量方法，不填写未经实践验证的数值。以下指标应在真实数据、候选算法和确定硬件环境中完成测试后冻结：

1. 边端采集与 WebRTC 视频播放性能。
2. 任务启动和结束耗时。
3. 各专题场景和综合监控允许组合的算法分析帧率。
4. 单帧处理延迟和前端结果展示延迟。
5. 结果有效时间和过期判定阈值。
6. 四类交通告警的生成延迟。
7. 故障持续判定时间和连续失败次数。
8. CPU、内存、GPU、显存和磁盘资源阈值。

冻结记录应包含测试环境、数据集版本、测量方法、测试结果、最终阈值和确认日期。冻结后的阈值应同步更新到需求、设计和测试文档。

## 11. 真实数据与算法验收要求

### 11.1 数据集划分

真实数据应按照开发、训练或适配、验证和测试用途分开管理，并以独立采集批次划分。测试数据不得参与模型训练、参数调整和阈值选择。

四类专题场景均应采集正常样本、目标事件样本和干扰样本。模型可以辅助产生初始标注，但验证集和测试集真值必须经过人工确认。

### 11.2 指标范围

| 业务功能 | 待冻结准确性指标 | 待冻结时效指标 |
| --- | --- | --- |
| 车牌识别 | 车牌检出率、完整字符正确率、白名单比对正确率、人工确认判定情况 | 车辆进入识别区域到结果显示的延迟 |
| 拥堵分析 | 区域车辆计数误差、拥堵等级判断正确率、热力图区域一致性 | 拥堵条件满足到告警生成的延迟 |
| 禁停跟踪 | 跟踪连续性、停留时间误差、告警正确率和重复告警控制 | 达到禁停阈值到告警生成的延迟 |
| 道路异常 | 障碍检出率、误报率、异常区域定位一致性和车道判断正确率 | 异常满足持续条件到告警生成的延迟 |
| 综合监控 | 各启用功能指标变化和多功能运行稳定性 | 结果刷新、事件延迟和资源占用 |

### 11.3 阈值确定流程

1. 完成真实数据采集和人工标注。
2. 划分训练或适配集、验证集和独立测试集。
3. 使用候选方案完成基线验证。
4. 调整模型、区域、规则和参数。
5. 冻结最终算法、配置和最低验收阈值。
6. 使用独立测试集执行最终测试。
7. 将测试结果写入系统测试报告。

阈值冻结后，不因最终测试结果不理想而临时降低。

开发优先级只用于安排实施顺序，所有 V2.0 范围内需求均需在最终验收前完成。

| 优先级 | 实施范围 |
| --- | --- |
| 高 | 登录与权限、移动 Web 边端、任务控制、WebRTC 视频上行与播放下行、实时监控、任务状态、基础数据持久化和任务视频留存 |
| 中 | 四类视觉业务、五类场景模板、设备与配置管理、交通告警、历史查询、系统监控和数据生命周期管理 |

## 13. 验收标准

## 14. 主要风险与控制要求

| 风险 | 影响 | 控制要求 |
| --- | --- | --- |
| 手机画面抖动或机位变化 | 区域配置失效，识别与事件判断不稳定 | 单次任务期间使用支架固定，任务前确认画面和区域 |
| 沙盘车牌过小或模糊 | 车牌字符无法可靠识别 | 使用闸机近景机位，规范车牌标识并验证图像质量 |
| WebRTC 连接失败 | 无法完成视频采集或播放 | 使用受信 HTTPS 域名，检查网络、信令和媒体连接状态 |
| 多功能组合占用过高 | 显存不足、分析延迟增加 | 建立经过实践验证的综合功能组合清单 |
| 磁盘空间不足 | 任务视频无法完整保存 | 任务前检查空间，运行中监控磁盘，执行 7 天清理策略 |
| 交通告警重复 | 告警列表和统计失真 | 按目标、区域和时间窗口合并持续事件 |
| 外部大模型生成事实错误 | 报告可信度下降 | 事实部分由云端固定生成，大模型只生成分析性正文并经过审核 |
| 外部大模型服务失败 | 报告无法生成 | 记录失败原因并允许重新发起生成 |
| 数据集划分不规范 | 测试结果虚高 | 以独立采集批次划分数据集，测试集不参与调参 |

| 历史与文件 | F-HIST、F-SYS | 历史查询、系统管理 | 视频、逐帧结果、长期档案 | A-MGT-05、A-DATA |
| 智能报告 | F-REPORT | 报告管理 | 报告版本、审核、PDF | A-REPORT |

## 16. 结论

V2.0 以真实视频、实时任务和三端协同为建设主线。移动 Web 边端负责现场采集，云端负责视频、算法、业务和数据处理，前端负责实时监控与平台管理。系统通过场景模板组织四类视觉业务，通过交通告警和历史数据形成业务闭环，并通过智能分析报告完成任务成果归纳与归档。

后续系统设计应以本文档为依据，明确模块、接口、数据结构、候选技术验证、部署方式和测试方法。完成真实数据与端到端实践后，应将最终算法选型和量化阈值同步补充到需求、设计和测试基线中。
