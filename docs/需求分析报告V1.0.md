# **云边端协同的智慧交通视觉感知系统需求分析报告 V1\.0**



## **1\. 引言**



### **1\.1 项目背景**



项目构建一个由手机、云服务器和前端展示端共同组成的智慧交通视觉感知平台，用于完成交通场景视频采集、传输、实时视频流 AI 分析、可视化展示、系统管理、设备管理、模型管理、监控展示和历史查询。



智慧交通场景中，车辆识别、车牌识别、拥堵状态判断、禁停区域监控和道路异常检测是典型视觉感知任务。传统单机脚本或离线视频分析难以体现真实系统中的设备接入、视频传输、云端分析和前端展示链路。因此，本项目将重点突出“云边端协同”特征，使边端手机、云端分析服务和前端展示端通过明确的数据协议形成稳定运行闭环。



### **1\.2 编写目的**



本文档用于明确项目 V1\.0 阶段的需求范围、用户角色、业务场景、功能需求、非功能需求、数据需求、界面展示需求、需求优先级、系统边界、验收标准和主要风险。本文档将作为后续系统设计方案 V1\.0、需求分析报告 V2\.0、系统实现、系统测试和验收演示的依据。

## **2\. 项目概述**

### **2\.1 项目目标**

本项目目标是构建一个面向现实交通状况（以学院 704 智慧交通沙盘为例）的云边端协同视觉感知系统。系统使用手机作为边端采集设备，使用电脑作为云端分析服务器，使用浏览器或另一台电脑作为前端展示端，实现从交通视频采集、视频传输、AI 分析、结果展示到历史查询的完整链路。

系统应优先保证可演示闭环，再逐步扩展多模型、多设备、多场景和更丰富的管理能力。V1\.0 阶段的核心目标包括：



1. 支持手机边端采集 交通状况（如704 智慧交通沙盘）视频，并通过实时视频流或视频段上传方式传输至云端。

2. 支持云端接收视频输入，并对视频进行车辆检测、车牌识别、拥堵分析、禁停跟踪和道路异常检测。

3. 支持前端展示实时视频、检测框、车牌结果、拥堵热力图、禁停告警、异常告警和系统状态。

4. 支持设备注册、设备在线状态、模型配置、系统状态监控和历史数据查询。

5. 在真实模型、网络或硬件受限时，支持使用短视频上传、规则判断和模拟分析结果，保证课程演示流程完整。

### **2\.2 系统组成**



系统由三类端组成：



|组成部分|角色定位|主要职责|
|---|---|---|
|边端手机|视频采集端|采集 交通状况（704 智慧交通沙盘），推送实时视频流或上传视频段，进行设备注册和心跳上报|
|云端分析服务|视频接入和 AI 分析端|接收视频流或视频文件，运行 AI 分析任务，生成检测结果、统计结果和告警事件|
|前端展示端|监控和管理端|展示实时视频与 AI 结果，提供设备管理、模型管理、系统状态、历史查询和统计页面|



### **2\.3 云边端协同目标**



本项目的云边端协同不应停留在概念层面，而应体现为可验证的数据链路：边端手机向云端传输视频流和设备心跳，云端对视频进行 AI 分析并向前端推送结构化结果，前端向云端提交模型配置、区域配置和历史查询请求。通过该闭环，系统能够模拟真实智慧交通平台中“边端采集、云端计算、前端展示”的协作方式。



## **3\. 用户角色与使用场景**



### **3\.1 用户角色**



|用户角色|使用目标|典型操作|
|---|---|---|
|边端设备使用者|使用手机采集交通视频|启动视频采集、推送视频流、上传视频段、查看设备连接状态|
|平台管理员|维护系统运行状态|查看设备在线状态、视频传输状态、CPU/GPU/内存状态和分析任务状态|
|算法/模型配置人员|配置 AI 分析任务|选择模型、设置置信度阈值、配置检测区域和禁停区域|
|监控查看人员|查看交通感知结果|查看车牌识别、拥堵热力图、禁停告警和道路异常告警|
||||



### **3\.2 典型演示场景**



默认演示场景为学院 704 智慧交通沙盘。典型演示流程如下：



1. 边端设备使用者使用手机对准 704 智慧交通沙盘中的道路、车辆、闸机或禁停区域进行视频采集。

2. 手机将实时视频流推送至云端服务，或在网络受限时上传短视频段作为输入。

3. 云端服务接收视频输入，启动 AI 分析任务，对车辆、车牌、拥堵状态、禁停行为和道路异常进行分析。

4. 前端展示端实时显示视频画面、检测框、车牌识别结果、白名单比对结果、拥堵热力图、车辆停留时长和异常告警。

5. 平台管理员查看设备在线状态、视频传输状态、模型运行状态和系统资源状态。

6. 评审人员通过历史查询页面查看车牌记录、拥堵趋势、违规停车记录和异常事件记录。

## **4\. 业务需求**



### **4\.1 车牌识别与白名单比对**



系统应对视频画面中的车辆进行车牌检测与识别，提取车牌号码、车牌位置和识别置信度，并与本地白名单车辆库进行比对。比对结果应输出“允许通行”或“未授权/需人工确认”等决策结果。课程项目不要求真实控制闸机，但前端应展示模拟闸机通行决策。



### **4\.2 车辆识别与拥堵热力图**



系统应检测视频画面中的车辆目标，统计不同道路区域内的车辆数量和密度等级，并动态生成拥堵热力图。拥堵热力图应能够直观展示 视频内容中道路区域的拥堵趋势，为监控展示和历史统计提供依据。



### **4\.3 禁停区域车辆跟踪**



系统应支持配置禁停区域，对进入禁停区域的车辆进行持续跟踪，记录车辆进入时间、当前位置、停留时长和跟踪标识。当车辆在禁停区域内停留超过预设阈值时，系统应生成长时间停留告警，并在前端突出显示。



### **4\.4 道路异常检测**



系统应检测道路区域内不应存在的异常物体或异常事件，例如障碍物、掉落物、静止异常目标等。发现异常后，系统应生成告警事件，标注异常位置、置信度、影响区域和受影响车道。V1\.0 阶段要求开放类别异常检测。

### **5\.2 云端视频接入功能**



|编号|需求描述|优先级|
|---|---|---|
|F\-CLOUD\-01|云端应支持接收或拉取边端手机实时视频流|P0|
|F\-CLOUD\-02|云端应支持接收视频段上传并进入分析流程|P0|
|F\-CLOUD\-03|云端应记录视频源状态，包括连接中、分析中、中断和结束|P1|

### **5\.3 云边端通讯功能**



|通讯链路|数据内容|推荐方式|优先级|
|---|---|---|---|
|边端手机 \-\> 云端服务|实时视频流、设备注册、设备心跳、视频源状态|RTMP/RTSP \+ REST API|P0|
|云端服务 \-\> 前端展示端|检测框、车牌结果、热力图数据、告警事件、系统状态|WebSocket \+ REST API|P0|
|前端展示端 \-\> 云端服务|模型配置、阈值配置、区域配置、历史查询请求|REST API|P1|
|云端服务 \-\> 边端手机|采集参数或设备控制指令|REST API 可选|P2|



系统应通过上述链路体现云边端协同。V1\.0 阶段必须完成边端到云端的视频输入链路、云端到前端的分析结果推送链路，以及前端到云端的查询和配置链路。



### **5\.4 实时视频流 AI 分析功能**



|编号|需求描述|优先级|
|---|---|---|
|F\-AI\-01|云端应能够从实时视频流中持续抽取视频帧|P0|
|F\-AI\-02|云端应能够对抽取帧进行车辆目标检测|P0|
|F\-AI\-03|云端应能够识别清晰车牌并生成白名单比对结果|P1|
|F\-AI\-04|云端应能够根据车辆检测结果生成拥堵等级和热力图数据|P1|
|F\-AI\-05|云端应能够对禁停区域车辆进行跟踪并计算停留时长|P1|
|F\-AI\-06|云端应能够识别疑似道路异常区域，或通过规则检测生成道路异常告警|P1|
|F\-AI\-07|云端应支持在模型不可用时输出结构化模拟结果|P0|



实时视频流 AI 分析不要求对每一帧完整推理，可采用抽帧分析方式，在 15\-30 FPS 的目标范围内平衡实时性和硬件压力。



### **5\.5 设备管理功能**



系统应支持边端设备注册、设备列表展示、在线状态监控、最近心跳时间展示、视频源状态展示和设备异常提示。设备信息应至少包括设备编号、设备名称、设备类型、在线状态、最近心跳时间、视频源地址和当前任务状态。



### **5\.6 模型管理功能**



系统应支持模型选择、模型运行状态展示、置信度阈值配置、检测区域配置、禁停区域配置和模型参数管理。V1\.0 阶段不要求在线训练模型，但应允许在真实模型和模拟结果模式之间切换，以保证演示稳定性。



### **5\.7 系统管理功能**



系统应展示云端服务的 CPU、GPU、内存使用率，以及视频传输状态、拉流状态、分析任务状态和 WebSocket 推送状态。若本地环境无法获取 GPU 信息，应显示“未检测到 GPU”或“GPU 监控不可用”，不影响系统主流程演示。



### **5\.8 监控展示功能**



系统应提供闸机监控、拥堵热力图、禁停区域监控、道路异常监控等展示视图，并支持根据演示需要切换监控画面、模型任务和展示方式。监控展示应突出实时性和可解释性，使评审人员能够理解 AI 结果如何产生。



### **5\.9 数据统计与历史查询功能**



系统应保存主要检测记录和告警事件，并支持按时间范围查询。历史统计应包括检测车牌数量、白名单命中情况、拥堵度变化趋势、违规停车数量、道路异常数量和异常位置分布。



## **6\. 界面与展示需求**



前端界面应服务于监控和管理场景，避免只做营销式页面。界面应清晰展示视频、AI 结果、系统状态和历史数据。



|页面/视图|展示内容|对应课程功能|优先级|
|---|---|---|---|
|实时监控大屏|视频画面、检测框、车牌结果、车辆数量、实时告警|监控展示、AI 分析结果展示|P0|
|闸机监控视图|车牌识别结果、白名单比对结果、是否允许通行|车牌识别|P1|
|拥堵热力图视图|区域车辆数量、密度等级、热力图叠加展示|车辆识别与拥堵热力图|P1|
|禁停区域监控视图|禁停区域框、车辆轨迹、停留时长、超时告警|车辆跟踪|P1|
|道路异常监控视图|异常物体位置、影响车道、告警级别|道路异常检测|P1|
|设备管理页面|手机设备列表、在线状态、心跳时间、视频源状态|设备管理|P0|
|模型管理页面|模型选择、阈值配置、区域配置、运行状态|模型管理|P1|
|系统状态页面|CPU/GPU/内存、视频传输状态、分析任务状态|系统管理|P1|
|历史查询页面|车牌记录、拥堵趋势、违规停车、异常事件查询|数据统计和历史查询|P1|



界面应支持桌面浏览器优先展示，并兼容常见笔记本屏幕分辨率。关键展示元素包括实时画面区域、指标卡片、事件列表、告警提示、地图或区域示意、趋势图表和配置表单。



## **7\. 关键问题需求决策**



### **7\.1 数据处理实时性与连续性平衡**



系统的视频输入与数据处理应以实时视频流为主，优先通过边端手机向云端推送 RTMP/RTSP 视频流，保证交通场景感知、AI 分析和前端展示具备实时性。云端服务接收视频流后，应采用连续拉流、按帧抽取和实时推理的方式生成车辆检测、车牌识别、拥堵热力图、禁停跟踪和道路异常检测结果，并通过 WebSocket 将结构化结果推送至前端展示端。



考虑到课程演示环境中可能存在网络波动、手机推流不稳定、云端算力不足或模型推理耗时较高等情况，系统应在实时性和连续性之间进行平衡。实时流处理过程中可采用抽帧分析策略，不要求对每一帧完整推理，而是在保证结果连续更新的前提下，将分析帧率控制在 10-15 FPS 范围内，以降低云端计算压力并提高系统稳定性。

当实时视频流中断或推流条件不满足时，系统应支持短视频段上传作为兜底输入。视频段上传不作为主方案，而是用于保证系统在异常环境下仍能完成“边端采集、云端分析、前端展示、历史记录”的演示闭环。视频段分析结果应尽量复用实时流分析的数据结构和展示方式，避免形成两套割裂的处理流程。

因此，V1.0 阶段的视频处理策略为：实时视频流优先，抽帧分析保障处理效率，视频段上传保障流程连续性。该策略既能体现智慧交通系统的实时感知能力，又能提高课程评审场景下的演示可靠性。



### **7\.2 数据传输方式**



系统应根据数据类型和实时性要求选择不同的传输方式，形成“视频流传输、业务接口调用、实时结果推送”三类通道。视频画面数据体量大、连续性强，主方案采用 RTMP/RTSP 进行传输，用于边端手机向云端推送或云端主动拉取实时视频流。该通道主要服务于实时视频接入和云端 AI 分析。



设备注册、设备心跳、模型配置、检测区域配置、禁停区域配置、白名单管理和历史数据查询等业务数据，采用 REST API 进行传输。REST API 适合请求响应式交互，便于前端管理页面调用，也便于后续进行接口测试、权限控制和文档维护。

云端 AI 分析结果、检测框、车牌识别结果、拥堵热力图数据、禁停告警、道路异常告警和系统运行状态等实时展示数据，采用 WebSocket 推送至前端展示端。WebSocket 能够减少前端轮询开销，使前端监控页面及时刷新分析结果和告警事件。

因此，V1.0 阶段的数据传输方式为：RTMP/RTSP 负责视频流，REST API 负责管理与查询，WebSocket 负责实时结果推送。三类通道共同支撑边端采集、云端分析和前端展示的协同运行。



### **7\.3 云边端如何通讯协同**



边端手机负责采集和上传视频，并向云端上报设备状态。云端服务负责视频接入、AI 分析、事件生成和数据持久化。前端展示端负责实时展示、管理配置和历史查询。三端之间通过明确协议协同工作，形成“边端采集 \-\> 云端分析 \-\> 前端展示 \-\> 配置反馈”的闭环。



V1\.0 阶段必须保证边端到云端、云端到前端、前端到云端三条链路可说明、可设计、可验收。云端对边端的复杂控制指令可作为 P2 扩展需求。



### **7\.4 实时视频流如何进行 AI 分析**



云端接收或拉取实时视频流后，应对视频帧进行抽取、预处理和 AI 推理，生成结构化分析结果。结构化结果应至少包含时间、来源设备、任务类型、目标位置、置信度、事件类型和前端可视化字段。



本节重点说明云端分析流程：视频流进入云端后，分析服务按配置频率抽取帧图像，调用车辆检测、车牌识别、目标跟踪、拥堵分析和异常检测模块，并将统一格式的结果通过 WebSocket 推送给前端，实现检测框、车牌、热力图、禁停时长和异常告警的实时刷新。抽帧频率和推理频率应服从 7.1 中的实时性与连续性平衡策略。



### **7\.5 处理时间效率如何设定**



V1\.0 阶段采用课程演示型实时指标。系统端到端延迟目标为 1\-2 秒，云端分析帧率目标为 15\-30 FPS。该指标重点服务课堂演示和系统稳定性，不承诺生产级小于 1 秒的低延迟能力。



若硬件算力不足，系统可降低抽帧频率、缩小输入分辨率、切换轻量模型或启用模拟结果模式，以保证前端展示连续性。



### **7\.6 业务场景模型选择**



系统的 AI 模型选择应围绕业务场景展开，而不是为所有任务套用同一种模型。V1.0 阶段应将车牌识别、车辆识别与拥堵热力图绘制、车辆跟踪、道路异常检测四个业务场景独立建模，并根据每类任务的输入、输出、评价指标和演示风险选择最合适的技术路线。

#### **7\.6\.1 车牌识别模型选择**

车牌识别采用 `YOLO + PaddleOCR + 车牌规则校验` 的技术路线。系统首先使用 YOLO 系列轻量检测模型从视频帧中定位车辆和车牌区域，输出车牌检测框、置信度和来源帧信息。随后云端对车牌区域进行裁剪，并根据画面质量进行灰度化、尺寸归一化、对比度增强、倾斜校正或透视矫正等预处理，以提高字符识别稳定性。

字符识别阶段严格采用 PaddleOCR。PaddleOCR 负责识别车牌中的中文省份简称、英文字母和数字字符，并输出完整车牌字符串及识别置信度。系统从 PaddleOCR 输出结果中提取数字字符，同时保留完整车牌号码，用于白名单比对、闸机通行决策展示和历史记录保存。由于车牌号码具有相对固定格式，系统应结合长度校验、字符类别校验和置信度阈值判断提高识别结果可信度。

V1.0 阶段不引入其他 OCR 主方案，避免技术路线发散。若车牌区域过小、模糊、遮挡或 PaddleOCR 置信度不足，系统应将识别结果标记为“需人工确认”或在演示模式下使用预设样例结果，但正式技术路线仍以 YOLO + PaddleOCR 为准。

#### **7\.6\.2 车辆识别与拥堵热力图绘制模型选择**

车辆识别与拥堵热力图绘制采用 `YOLO 车辆检测 + 区域密度规则 + 时间窗口统计` 的技术路线。YOLO 负责识别画面中的车辆目标并输出检测框、类别和置信度，拥堵热力图不单独训练热力图模型，而是根据车辆检测结果、区域配置和时间窗口统计生成。

热力图应采用分区密度热力图方案。系统先将 704 智慧交通沙盘道路划分为入口区、路口区、闸机等待区、转弯区和禁停区域周边等若干检测区域，再根据车辆检测框中心点所属区域统计车辆数量和区域密度。拥堵等级可根据“区域车辆数 / 区域容量”计算得到，并结合持续时间阈值降低单帧误检造成的抖动。

前端展示时应采用视频底图层、热力覆盖层和信息标注层三层结构：视频底图层展示实时画面，热力覆盖层使用半透明色块表达拥堵等级，信息标注层展示区域名称、车辆数、趋势箭头和持续时间。历史统计中也应保存各区域拥堵等级变化，用于绘制拥堵趋势曲线。该方案能够让热力图同时表达空间位置、拥堵程度、变化趋势和统计依据，而不是仅提供装饰性颜色效果。

#### **7\.6\.3 车辆跟踪模型选择**

车辆跟踪采用 `YOLO 检测 + ByteTrack 跟踪 + 禁停区域规则` 的技术路线。YOLO 提供逐帧车辆检测框，ByteTrack 根据检测框在连续帧中的位置变化为车辆分配跟踪 ID，禁停区域规则负责判断车辆是否进入禁停区域并计算停留时长。

该方案比单纯逐帧检测更适合禁停判断。系统应记录车辆跟踪 ID、进入禁停区域时间、当前位置、停留时长和告警状态。当同一车辆在禁停区域内停留超过预设阈值时，系统应触发长时间停留告警，并在前端显示车辆轨迹、停留时长和告警级别。若跟踪 ID 因遮挡或画面抖动短暂丢失，系统可通过短时间缓冲窗口降低误告警概率。

#### **7\.6\.4 道路异常检测模型选择**

道路异常检测不应简单等同于常规目标检测。由于本项目场景中的异常物体可能未知，难以提前穷举所有异常类别，因此该业务场景应采用 `FastFlow 异常检测模型 + 规则检测兜底` 的技术路线，而不把目标检测模型作为道路异常检测的主方案或兜底方案。

FastFlow 的目标是通过正常道路样本学习道路画面的正常特征分布，并在实时视频帧中发现“不符合正常道路状态”的疑似异常区域。相比更重的异常检测方法，FastFlow 更贴合本项目 15-30 FPS 的课程演示型实时要求，能够输出异常分数和异常热力区域，便于前端标注异常位置、影响区域和告警等级。该思路更符合未知异常检测的业务目标，适用于异常物体类型无法提前确定的情况。

考虑到 V1.0 阶段以 704 智慧交通沙盘演示为主，FastFlow 对正常样本质量、摄像头稳定性和阈值配置仍有要求，系统应保留规则检测作为兜底方案。规则检测可基于固定摄像头视角、道路区域掩膜、背景差分、静止区域变化、异常面积阈值和持续时间阈值判断疑似异常。规则检测不负责识别具体异常物体类别，而是用于在 FastFlow 置信度不足、模型不可用或推理延迟过高时，保证系统仍能发现“道路上出现异常区域”并完成告警展示。

#### **7\.6\.5 模型效果评估与选择原则**

系统不应以单一指标判断模型优劣，而应综合准确性、实时性、稳定性、可解释性、部署成本和演示风险选择当前场景下效果最好的模型组合。若多个模型效果接近，应优先选择能够在现有硬件条件下稳定达到 5-15 FPS、端到端延迟 1-3 秒，并能支撑前端演示闭环的方案。

|业务场景|关键准确性指标|实时性指标|稳定性关注点|可解释输出|
|---|---|---|---|---|
|车牌识别|车牌字符串正确率、数字识别正确率、白名单比对正确率|单帧车牌检测与 OCR 耗时|车牌大小、角度、模糊、遮挡|车牌框、完整车牌号、数字字符、OCR 置信度|
|车辆识别与拥堵热力图|车辆检测准确率、漏检率、区域车辆数误差|车辆检测 FPS、热力图刷新频率|小目标、遮挡、区域边界误判|检测框、区域车辆数、密度等级、趋势变化|
|车辆跟踪|ID 保持率、跟踪丢失次数、禁停告警正确率|跟踪 FPS、告警延迟|遮挡、车辆交叉、短时丢帧|车辆轨迹、跟踪 ID、停留时长、告警状态|
|道路异常检测|异常召回率、误报率、异常区域定位准确性|异常检测耗时、告警延迟|光照变化、背景变化、未知异常形态|异常区域、异常分数、影响车道、持续时间|

V1.0 阶段的模型准确度要求限定为课程演示准确度：在光照稳定、摄像角度固定、目标无遮挡或轻微遮挡的 704 沙盘条件下，系统应能够稳定输出主要车辆目标、清晰车牌结果、拥堵趋势、禁停停留时长和道路异常告警。不承诺复杂真实道路、极端光照、严重遮挡和生产级全场景准确率。



## **8\. 非功能需求**



|类别|需求描述|
|---|---|
|实时性|端到端延迟目标为 1\-2 秒，分析帧率目标为 15\-30 FPS|
|通讯稳定性|设备心跳、视频源状态和 WebSocket 连接状态应可监控，异常时应提示|
|AI 准确度|在清晰画面下达到课程演示可用，允许规则和模拟兜底|
|可演示性|即使实时流、GPU 或模型不稳定，也应通过视频段和模拟结果保证演示闭环|
|可扩展性|系统应支持后续接入更多设备、模型和交通场景|
|可维护性|功能模块应清晰划分，接口和数据结构应便于测试和修改|
|数据持久化|设备、白名单、模型配置、告警事件和历史统计应尽量保存到本地数据库|
|安全性|管理接口应预留基本访问控制设计，避免未授权修改模型和配置|
|兼容性|系统应兼容无 GPU 或模型文件不完整的演示环境|



## **9\. 数据需求**



### **9\.1 设备数据**



设备数据应包含设备编号、设备名称、设备类型、注册时间、在线状态、最近心跳时间、视频源地址和当前任务状态。



### **9\.2 视频源数据**



视频源数据应包含视频源编号、设备编号、输入类型、视频流地址、视频文件路径、接入时间、断开时间和传输状态。



### **9\.3 AI 分析结果数据**



AI 分析结果应采用结构化格式，至少包含时间戳、设备编号、任务类型、目标类别、目标位置、置信度、跟踪编号和可视化字段。目标位置可使用边界框坐标、区域编号或图像坐标表示。



### **9\.4 告警事件数据**



告警事件应包含事件编号、事件类型、发生时间、来源设备、关联目标、事件位置、告警级别、处理状态和备注信息。事件类型包括未授权车牌、拥堵告警、禁停超时告警和道路异常告警。



### **9\.5 模型配置数据**



模型配置应包含模型名称、模型类型、运行状态、置信度阈值、检测区域、禁停区域、拥堵阈值、停留时间阈值和是否启用模拟模式。



### **9\.6 历史统计数据**



历史统计数据应支持按时间范围统计车牌识别数量、白名单命中数量、拥堵等级变化、违规停车数量、道路异常数量和异常位置分布。



## **10\. 需求优先级建议**



### **10\.1 优先级定义**



|优先级|含义|说明|
|---|---|---|
|P0 必须完成|最小可演示闭环必需能力|缺少后会影响立项后系统主线成立|
|P1 优先完成|课程核心功能和主要展示能力|应在中期或验收前尽量完成|
|P2 可选增强|展示效果、创新性或扩展能力|有助于加分，但不影响基础闭环|



### **10\.2 分级建议**



|优先级|需求范围|建议说明|
|---|---|---|
|P0|手机视频输入、云端视频接收、实时或视频段 AI 分析、前端实时展示、基础告警、设备在线状态、模拟结果兜底|保证系统形成“边端输入 \-\> 云端分析 \-\> 前端展示”的最小闭环|
|P1|车牌识别、拥堵热力图、禁停跟踪、道路异常检测、模型配置、历史查询、系统状态监控|覆盖课程核心功能，提高评审完整度|
|P2|多设备接入、多模型切换、GPU 详细监控、统计图表优化、云端对边端反向控制、更多道路场景兼容|用于中期和验收阶段增强系统效果|



## **11\. 系统边界与约束**



1. 系统不要求真实控制闸机，只需在前端展示车牌白名单比对和模拟通行决策。

2. 系统不承诺生产级实时性，V1\.0 阶段采用端到端 1\-2 秒延迟和 15\-30 FPS 分析帧率作为演示目标。

3. 系统不强制训练自有大模型，优先使用轻量化检测模型、OCR 工具、跟踪算法和规则逻辑组合实现。

4. 系统不提交大体积原始视频和大模型权重，应通过 README、下载说明、占位文件或配置说明管理外部资源。

5. 系统首选 704 智慧交通沙盘场景，后续可兼容校园道路或校外道路场景。

6. 系统允许在 GPU、真实模型或网络环境不稳定时使用视频段上传和模拟分析结果作为演示兜底。

## **12\. 验收标准**



|验收项|验收标准|
|---|---|
|边端采集|手机能够采集视频，并优先完成实时视频流推送；在推流异常时能够使用视频段上传作为兜底输入|
|云边端通讯|设备能够注册、上报心跳，云端能够显示设备在线状态和视频源状态|
|视频接入|云端能够优先接收实时视频流并启动分析任务；实时流不可用时能够接收视频段并复用同一套分析和展示流程|
|实时 AI 分析|云端能够输出车辆、车牌、拥堵、禁停或异常中的至少一种真实或模拟分析结果|
|前端展示|前端能够展示视频画面、检测结果、告警事件和关键指标|
|车牌识别|系统能够展示车牌识别结果和白名单比对结果|
|拥堵热力图|系统能够按区域展示车辆数量、密度等级或热力图效果|
|禁停跟踪|系统能够展示禁停区域、车辆停留时长和超时告警|
|道路异常|系统能够展示异常位置、告警类型和影响区域|
|系统管理|系统能够展示 CPU、GPU 或内存状态，以及视频传输和分析任务状态|
|历史查询|系统能够查询历史车牌记录、拥堵趋势、违规停车或异常事件|
|准确度演示|在 704 沙盘清晰画面下，系统能够稳定输出课程演示可用的分析结果|



## **13\. 风险与应对**



|风险|影响|应对措施|
|---|---|---|
|实时视频流不稳定|影响云端分析和前端展示|保留视频段上传作为兜底方案|
|云边端通讯中断|影响设备在线状态和结果推送|增加设备心跳、连接状态展示和重连提示|
|模型效果不足|影响识别准确度和演示可信度|使用轻量模型、调整阈值、限定场景、保留模拟结果|
|AI 分析准确度不稳定|影响评审对算法效果的判断|明确课程演示准确度边界，使用规则逻辑和人工样例配合演示|
|硬件算力不足|影响帧率和延迟|降低分辨率、降低抽帧频率、使用轻量模型或模拟模式|
|704 场景数据不足|影响模型适配|使用公开模型、少量样例视频、预设区域和规则方法降低数据依赖|
|前端展示信息过多|影响演示理解|按实时监控、设备管理、模型管理和历史查询分页面组织|
|时间周期较短|影响功能完整度|按 P0/P1/P2 分级推进，优先完成可演示闭环|



## **14\. 后续迭代方向**



V2\.0 阶段应在 V1\.0 基础上进一步细化接口定义、数据结构、系统设计方案、数据库表结构、模型配置参数、测试指标和演示流程。后续可重点推进以下方向：



1. 明确 REST API 和 WebSocket 消息格式。

2. 细化设备管理、模型管理和历史查询的数据表设计。

3. 完善实时视频流接入和视频段上传的统一处理流程。

4. 明确真实模型和模拟模型的切换机制。

5. 增加前端页面原型和系统演示脚本。

6. 完善系统测试报告中的接口测试、算法测试、前端流程测试和联调测试。

## **15\. 结论**

本项目 V1\.0 需求以现实交通情况为主要场景，以云边端协同为系统核心，以实时视频流 AI 分析为主要技术主线，以课程演示可用为阶段目标。系统需求覆盖边端采集、云端分析、前端展示、设备管理、模型管理、系统管理、历史查询和数据统计，并通过 P0/P1/P2 优先级划分保证短周期内优先完成可演示闭环。

后续系统设计和开发应围绕“手机采集视频、云端实时分析、前端展示结果、历史数据可查”的主线推进，避免偏离为单机脚本或纯前端展示页面。
